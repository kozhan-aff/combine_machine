# VPN Affiliate Portfolio — План проекта

Система полного цикла: поиск и выкуп доменов → скоринг качества → провижн на Cloudflare/VPS → генерация и публикация контента → простановка реф-ссылок → мониторинг и жизненный цикл. Вертикаль — VPN (легальная, вечнозелёная). Управление — Docker-приложение с панелью.

---

## 0. Кратко (TL;DR)

Строим **систему управления портфелем качественных affiliate-сайтов** в VPN-нише, а не фабрику тонких сборников в объёме. Движок — замкнутый цикл из 6 модулей. Стартуем с модуля скоринга доменов (полезен сам по себе, ноль риска, на **бесплатном стеке** — Wayback/РКН/SearXNG/OpenPageRank, без платных API), затем сквозная петля на одном домене = MVP, дальше — автоматизация и масштаб.

Главный рычаг проекта — не уникальность продукта (её сейчас нет ни у кого, всё копируется за выходные), а **дистрибуция через SEO**, которой у мелких конкурентов нет. Это ровно то, что есть у нас и почти ни у кого в связке с умением кодить.

## 2. Проектный принцип и контекст рисков (прочитать один раз)

Одно решение задаёт всю архитектуру, поэтому фиксируем его явно и дальше не возвращаемся.

**Мы строим портфель немногих качественных сайтов, а не рой тонких сборников.** Это не вопрос вкуса — это то, что определяет, проживёт система больше месяца или нет. Данные 2026 года:

- Мартовский core-апдейт Google ударил по affiliate-сайтам сильнее любой другой категории — **71% просели**. Хуже всего — обзоры без реального тестирования продукта.
- **Scaled content abuse** — теперь явная, активно применяемая спам-политика. Массовый тонкий AI-контент без ценности → падения трафика **50–80%**.
- **Expired domain abuse** назван прямой целью июньского спам-апдейта 2026. Ссылочный «сок» дропнутого домена, на который исторически опиралась эта модель, гасится намеренно.

Что из этого следует для дизайна (встроено в модули ниже):

1. **Домены** выбираем за *чистую историю и тематическую релевантность*, а не за прошлое DR. Не ждём буста от «сока» — берём домен, чтобы он не тянул назад, а не чтобы вытянул вперёд.
2. **Контент** = AI-черновик + **обязательная человеческая редактура** + реальный *information gain* (для VPN: замеры скорости, тесты обхода гео-блоков, данные из приложений). Качество текста — вся игра разом, на нём всё держится.
3. **Топикальная авторитетность** важнее объёма. Лучше 5 сайтов с реальной глубиной, чем 100 однотипных.

AI сам по себе не наказывается — наказывается тонкий AI в объёме. Переменная, определяющая исход, — редакторский слой поверх черновика.

## 3. К чему стремимся (North Star)

Полностью собранная система — это Docker-приложение с панелью, через которое один человек управляет портфелем VPN-сайтов по замкнутому циклу:

- **находит** освобождающиеся/освободившиеся домены из нескольких источников,
- **скорит** их по качеству и чистоте истории автоматически,
- ставит годные **в очередь на выкуп** (финальное подтверждение — за человеком),
- **разворачивает** сайт: регистрация → Cloudflare → хостинг на VPS,
- **генерирует** контент через LLM с обогащением реальными данными и редакторским чекпоинтом,
- **публикует**, сабмитит в Search Console и **следит за индексацией**,
- **отслеживает перформанс** по портфелю, отбраковывает неудачное с 301 на рабочее, итерирует.

«Готово» = петля крутится с минимальным ручным вмешательством (только решение о покупке и редактура контента), а панель даёт видимость всего портфеля: какие домены в работе, что проиндексировано, что приносит трафик, что пора отбраковать.

## 4. Архитектура — 6 модулей

### Модуль 1. Domain Intelligence (ядро, стартуем отсюда)
- **Вход:** дроп-лист backorder.ru (публичный JSON, без auth; `links=N` доноров, `tomorrow=1`) [+ опц. др. источники].
- **Обработка (free-стек):** OpenPageRank (грубый DR-прокси, пре-фильтр) + Wayback-парсинг на прошлое использование (флаг адалт/фарм/казино/спам, смена тематики, возраст) + РКН (antizapret) и Spamhaus (свой резолвер) — hard-reject + SearXNG `site:` (indexed_echo). Платный ссылочный профиль (RD/анкоры/live) — опц. стадия, когда появится бюджет / через Ahrefs MCP на review.
- **Выход:** отскоренный шортлист в БД, видимый в панели.
- **Почему первый:** приносит пользу немедленно, ещё до траты денег; работает на бесплатном стеке; ноль риска.

### Модуль 2. Acquisition (полуручной осознанно)
- **Вход:** одобренный шортлист из Модуля 1.
- **Обработка:** API оптимизатора (регистрация освободившихся) + API бэкордеров (ещё не отпустившиеся) + очередь на выкуп.
- **Ключевое:** финальное «купить» подтверждает человек. Денежные решения не отдаём на автопилот — и здравый смысл, и защита от дорогой ошибки в скрипте.
- **Выход:** купленный домен → передаётся в Модуль 3.

### Модуль 3. Provisioning
- **Вход:** купленный домен.
- **Обработка:** Cloudflare API (зона, DNS, SSL) → провижн на VPS (vhost через aaPanel API или напрямую nginx) → запись сайта в центральную БД.
- **Выход:** поднятый хостинг за Cloudflare, готовый под контент.

### Модуль 4. Content Pipeline (make-or-break)
- **Вход:** поднятый сайт + выбранный шаблон/структура (можно от конкурента из выдачи).
- **Обработка:** скаффолд структуры → генерация через LiteLLM `192.168.1.77:4000` (`mistral`=mistral-large ради качества; `ollama/*` для черновиков) → **обогащение реальными данными вертикали** (фиды/API VPN-провайдеров: замеры, характеристики) → **чекпоинт ручной редактуры** → вставка реф-ссылок с раскрытием (disclosure).
- **Выход:** готовый к публикации контент.
- **Примечание:** самый трудоёмкий и самый решающий модуль. Здесь живёт принцип из §2.

### Модуль 5. Publish & Monitor
- **Вход:** готовый контент.
- **Обработка:** деплой на сайт → **проверка индексации** (v1: ручной `site:` через SearXNG; GSC исключён из v1, позже — Bing WMT для автосабмита sitemap + инспекции).
- **Выход:** опубликованный сайт + поток данных по индексации в панель.
- **Примечание:** мониторинг индексации — ранний сигнал проблем, раньше чем упадёт трафик.

### Модуль 6. Lifecycle (не в MVP)
- Трекинг перформанса по сайтам, отбраковка неудачных с 301 на рабочие (нормальная гигиена портфеля), итерации и миграции.

## 5. Стек

- **Бэкенд:** Python + FastAPI — задача по природе это склейка API + данные + LLM; Python удобнее всего, и ты вайб-кодишь через Claude Code.
- **БД:** PostgreSQL — домены, сайты, скоринг, статусы, контент, история индексации.
- **Очереди/асинхронщина:** Celery или RQ (пуллы метрик, чек индексации). Для MVP хватит cron/APScheduler.
- **Панель:** для MVP — FastAPI + HTMX прямо в backend (BUILD_SPEC §2); отдельный Nuxt — позже, если понадобится.
- **Оркестрация:** docker-compose — сервисы `backend`, `worker`, `db` (+ `panel`, `redis` позже). LiteLLM/SearXNG/A-Parser — **внешние**, на боксе 192.168.1.77.
- **Секреты:** api_key оптимизатора, Cloudflare-токен, aaPanel api_sk, backorder login, A-Parser-пароль — в `.env`/secrets, не в коде. LiteLLM/SearXNG — без ключей.
- **Внешние интеграции:** backorder.ru · optimizator.ru · Cloudflare API · aaPanel API · **LiteLLM** (контент) · **SearXNG** (SERP) · **A-Parser** (whois/keywords) · Wayback · РКН · Spamhaus · OpenPageRank. *(Ahrefs API — нет; GSC — исключён из v1.)*

## 6. Дорожная карта (по фазам)

### Фаза 0 — Фундамент
Репозиторий, скелет docker-compose, схема Postgres, `.env`/секреты, smoke-тесты коннективности (free/локальные без кредов: wayback, rkn/antizapret, searxng, litellm, openpagerank, backorder-discovery; с кредами позже: cloudflare, aapanel, optimizator, a-parser). Цель — убедиться, что до каждого сервиса достучались, прежде чем строить логику.

### Фаза 1 — MVP (сквозная петля на одном домене)
- **Модуль 1 полностью** — скоринг доменов на free-стеке (строим первым, полезен сам по себе, без платных API).
- **Модули 3–4–5 на ОДНОМ домене** — доказать цикл «поднять сайт → налить качественный контент → опубликовать → увидеть индексацию».
- **Покупку в MVP делаем руками** — авто-выкуп не нужен для доказательства петли.
- **Критерий готовности MVP:** один реальный сайт поднят системой, с редактированным контентом, опубликован, страницы в индексе; скоринг доменов работает и выдаёт шортлист.

### Фаза 2 — Полуавтоматизация
- Модуль 2 (обёртка Acquisition + очередь с подтверждением).
- Модуль 3 в полную автоматизацию (домен на вход → хостинг на выходе без ручных шагов).
- Система шаблонов в Модуле 4.

### Фаза 3 — Контент-качество как система
- Зрелость Модуля 4: обогащение из фидов VPN-провайдеров, редакторский воркфлоу, disclosure-механика.
- Дашборд мониторинга в Модуле 5 (индексация + первые сигналы трафика).

### Фаза 4 — Масштаб портфеля + Lifecycle
- Модуль 6: трекинг перформанса, отбраковка с 301, миграции.
- Управление мультисайтом из панели.

### Полная версия (North Star)
Замкнутый цикл из §3 крутится с минимальным ручным вмешательством.

**Честно про сроки:** с вайб-кодингом Модуль 1 — несколько дней. Петля провижн+контент+паблик на одном домене — ещё неделя-две. Упаковка в Docker с панелью — сверху. **Рабочее MVP по ядру — несколько недель фокусной работы, не выходные:** много внешних API, каждую интеграцию надо подключить и обвязать обработкой ошибок.

## 8. Риски и снижение

| Риск | Снижение |
|---|---|
| Google: scaled content / expired domain abuse | Контент качество-first + редактура; домены за чистую историю, не за «сок»; топикальная авторитетность вместо объёма |
| Один VPS/IP — footprint и точка отказа | Cloudflare маскирует origin; сайты независимые, **не** перелинкуем ради веса; при росте — второй бокс |
| ToS партнёрок банят типы трафика | Disclosure и разрешённые источники встроены в пайплайн; читать условия каждой программы до захода |
| Контент-бутылочное горло | Редакторский чекпоинт; LiteLLM `mistral-large` для чистовика (`ollama` для черновиков); реальные тестовые данные |
| Насыщенность ниши (Nord/Surfshark) | Узкие углы (страна/юзкейс/протокол), проверенные через SearXNG/free-стек, а не «best VPN» в лоб |
| Депортизация/деиндексация | Мониторинг индексации (v1 — ручной `site:`) как ранний сигнал; не складывать все домены в один паттерн |

## 10. Ближайшие шаги

1. **Старт — Модуль 1 (Domain Intelligence).** Приносит ценность немедленно, работает на бесплатном стеке (discovery публичный, метрики/SERP/история — free), ничего не стоит и ничем не рискует. Даёт рабочий инструмент скоринга, даже если остальной комбайн решишь не достраивать.
2. Параллельно — **Фаза 0** (скелет docker-compose + схема БД + smoke-тесты API), чтобы Модуль 1 сразу лёг в каркас.
3. Когда скоринг выдаёт вменяемый шортлист — переходим к петле на одном домене (Модули 3–4–5) = MVP.
