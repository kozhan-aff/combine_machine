# Спека 4 — LLM-критик редактуры: дизайн

**Статус:** автономная итерация (без диалога, по прецеденту Треда D) — пользователь
явно запросил продолжать автономно, не блокируясь на живом боксе. Дизайн зафиксирован
здесь, решения задокументированы с обоснованием, чтобы их можно было пересмотреть
постфактум.

## Проблема

`content.generate_site()` пишет черновик одним LLM-вызовом на страницу. Дальше
человек ОБЯЗАН вручную перевести `draft -> edited` (`mark_edited`, хард-гейт,
CLAUDE.md правило 1) — без этого публикация невозможна. Но перед тем как открыть
черновик, человек не знает, стоит ли вообще смотреть: пустая генерация уже отловлена
(`if not body.strip(): continue`), но откровенно слабый черновик (не в тему офферу,
без конкретных фактов вертикали, не на том языке, без disclosure) — нет. Человек
тратит время на чтение того, что и так придётся почти полностью переписывать.

## Цель

Второй, более дешёвый LLM-вызов — «критик» — оценивает черновик ПЕРЕД тем, как
человек его откроет, и показывает баллы/проблемы на экране редактуры. Экономит время
человека на очевидно слабых страницах. **Гейт редактуры НЕ трогается ни при каких
обстоятельствах** — `mark_edited` остаётся единственным путём в `edited`, критик не
может ни одобрить, ни отклонить публикацию сам.

## Что НЕ делаем (сознательно, чтобы не расширять скоуп)

- **Не автоматизируем `draft -> edited`.** Даже отличный балл критика не двигает
  статус — это прямое нарушение хард-гейта 1, обсуждению не подлежит.
- **Не запускаем критика автоматически при каждой генерации.** Раздельная кнопка на
  экране редактуры (`POST /pages/{id}/critique`) — токены LLM не жгутся молча на
  каждый прогон свипа (конвенция «беречь LLM-токены»); критик — инструмент ПОМОЩИ
  человеку, вызывается им по требованию.
- **Не блокируем сохранение/mark_edited, если критика не было или она провалилась.**
  Критик — чисто advisory-слой, отсутствие/ошибка оценки не создаёт нового состояния,
  которое надо чинить, чтобы двигаться дальше (в отличие от `blind_reason` в M1-
  скоринге, где непроверенная история — реальный риск денег; здесь риск другой: слабый
  текст, который человек и так увидит при открытии страницы).

## Модель данных

Миграция `0023_page_critic` — три nullable-колонки на `Page` (симметрично
`index_status`/`index_checked_at`, тот же паттерн «поле + когда проверено»):

```python
critic_score: Mapped[float | None] = mapped_column(Float)        # 0.0–1.0, None = не оценено
critic_notes: Mapped[dict | None] = mapped_column(JSON)          # {"issues": [str, ...]}
critic_checked_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
```

`critic_notes["issues"]` — список коротких русских строк-замечаний (не структура,
не enum: критик пишет то, что видит, человек читает как обычный список). Никакого
отдельного поля-вердикта («годится»/«не годится») — балл + список замечаний, решение
всегда за человеком.

## Логика оценки — `services/content_critic.py`

```python
def critique_page(page_id: int) -> dict:
    """Оценить черновик страницы вторым LLM-вызовом. Пишет critic_score/critic_notes/
    critic_checked_at на Page. НЕ трогает status. Возвращает {"score": float|None,
    "issues": [str], "error": str|None} — error != None значит оценка не удалась
    (транспорт/парсинг), score/issues в этом случае None/[]."""
```

Критерии оценки в промпте критика (пять пунктов, зеркалят инварианты качества из
CLAUDE.md — не придуманы заново, взяты из уже существующих требований к контенту):
1. Тематическая связность с офером/брендом страницы (`page.offer_id` → бренд).
2. Есть ли конкретные факты вертикали (цифры/сравнения), а не только общие фразы.
3. Язык страницы соответствует `page.lang`.
4. Есть disclosure (пометка о партнёрской ссылке).
5. Текст не выглядит как «вода» (общие AI-фразы без содержания).

**Формат ответа LLM — НЕ строгий JSON.** LiteLLM/mistral-large на этом боксе никогда
не проверялся на структурированный вывод (LLM живьём НЕ ДОСТУПЕН в этой итерации —
`192.168.1.77:4000` за тем же боксом, что и A-Parser/панель, все порты закрыты все три
проверки). Гадать формат — повторить ровно ту ошибку, что дважды стоила проекту дорого
(whois-возраст, cctld-дата дропа). Поэтому:
- Промпт просит ПРОСТОЙ построчный формат: первая строка `БАЛЛ: <0-100>`, дальше
  строки, начинающиеся с `- ` — замечания.
- Парсер (`_parse_critique(text: str) -> dict`) читает regex на первую строку
  (`r"БАЛЛ:\s*(\d+)"`), если не находит — `score=None` (не гадать, не подставлять 0
  как «оценено плохо»); собирает `- `-строки как issues, без вопросов — пустой список
  не ошибка. Никаких исключений наружу из парсера — сплошной best-effort разбор текста.
- Если LLM вернул пустую строку (см. `LlmClient.complete` — уже возвращает `""` на
  фильтр/blocked) — `error="пустой ответ LLM"`, score/issues пустые, критик честно
  говорит «не смог оценить», а не подсовывает нулевой балл как результат.
- **Первый живой прогон обязан свериться с реальным форматом ответа** и при
  необходимости поправить парсер/промпт — ЭТО НЕ СДЕЛАНО в этой итерации (тот же
  принцип честности, что для whois/cctld/reg.ru/sweb в истории проекта). Пометить в
  докстрине `content_critic.py` явно.

## UI — `page_edit.html` + панельный роут

`POST /pages/{id}/critique` (панель, `panel.py`) — синхронный вызов (одна страница,
один LLM-запрос, не фоновая задача как Discovery/Score — счёт на секунды, не минуты),
редиректит назад на `/pages/{id}` с `msg`/`err` через уже существующий `_back()`.

На `page_edit.html` — блок над формой сохранения:
- Кнопка «Оценить черновик» (не блокирует форму сохранения, отдельная).
- Если `page.critic_checked_at` не пусто — показать балл (0-100, округлённый от
  `critic_score * 100`) и список `critic_notes.issues` как обычный `<ul>`.
- Подпись под баллом (шильдик, прогрессивный принцип проекта): «критик — подсказка,
  не решение; сохранить как edited можно независимо от балла».

## Хард-гейты — явное подтверждение

- **Денежный гейт** — не задет вообще, этот код не про M2.
- **Редактурный гейт** — `mark_edited`/`page.status` НИ ОДНОЙ строкой этой фичи не
  трогаются. `critique_page()` физически не имеет доступа менять `status` (не
  импортирует его, не пишет). Ревьюер (`combine-reviewer`) обязан явно проверить это
  на каждой задаче SDD-прогона.
