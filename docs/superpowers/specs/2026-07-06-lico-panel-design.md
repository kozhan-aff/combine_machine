# Спек 2 «Лицо» — локализация + прогрессбар + умный /diag + полиш

**Статус:** дизайн одобрен (2026-07-06), готов к плану.
**Родитель:** «Комбайн» — машина полного цикла VPN-портфеля. Это второй суб-спек:
после «Мозга M1» (Спек 1, приобретаемость — влит) причёсываем **лицо** панели.
Автономия (тумблеры, шедулер) — отдельный Спек 3, здесь НЕ трогаем.

**Цель:** панель говорит по-русски одним голосом, длинные задачи показывают реальный
визуальный прогресс, диагностика читается как дашборд здоровья, мелкие косметические
долги закрыты. Дизайн-контракт (светлая CMS, шильдик, CSS-переменные `base.html`) —
неизменен, только доиспользуется.

## §A. Локализация через центральную display-мапу

**Проблема:** англицизмы (Discovery/Score/discovered/scored/reject-коды) размазаны по
шаблонам как строковые литералы. Нет единой точки правды → правки рассинхронизируются.

**Решение:** новый модуль `backend/app/services/labels.py` — три словаря + регистрация
Jinja-фильтров. Одна точка правды, тестируемая.

```
STATUS_RU = {
  "discovered": "найден", "scored": "оценён", "approved": "одобрен",
  "rejected": "отклонён", "purchasing": "в очереди", "purchased": "куплен",
  "provisioning": "поднимается", "content": "контент", "draft": "черновик",
  "edited": "отредактирован", "published": "опубликован", "live": "живой",
}
REJECT_RU = {
  "low_rd": "мало доноров", "feed_flag": "флаг источника", "too_young": "моложе порога",
  "rkn": "реестр РКН", "blacklist": "блэклист", "history_dirty": "грязная история",
  "low_score": "низкий скор", "not_acquirable": "нельзя купить",
}
LANE_RU = { "bid": "ставка", "free": "свободный" }
```

- Функции `status_ru(v)`, `reject_ru(v)`, `lane_ru(v)` — возвращают перевод или **сырое
  значение как фолбэк** (неизвестный/None ключ не роняет шаблон, показывает как есть).
- Регистрируются как Jinja-фильтры там, где создаётся `Jinja2Templates`
  (`app/api/panel.py:27`, объект `templates`): `templates.env.filters["status_ru"] =
  status_ru` и т.д.
- **Правило показа reject:** бейдж/ячейка статуса показывает **русскую фразу + мелкий
  `<code>сырой_код</code>` рядом** (например «нельзя купить `not_acquirable`») — точность
  для дебага сохраняется, читаемость появляется. Легенда — та же пара.
- **Статусы-бейджи:** текст бейджа = `d.status|status_ru`, CSS-класс `b-{{ d.status }}`
  остаётся на сыром значении (цвет привязан к enum, не к переводу).
- **Лейн-колонка и тултип:** `d.lane|lane_ru` вместо inline-тернаров `'ставка' if …`.
- **Станции/кнопки/проза/прогресс:** **Discovery → «Поиск дропов», Score → «Запуск
  проверки»** (глагол действия). Технические подписи модулей (`M1a · 4 источника`)
  остаются. В прозе разъясняющей воронку — по-русски, но термины воронки (RD, whois,
  score) остаются как термины (они в легенде объяснены).
- **НЕ переводим собственные имена сервисов:** Cloudflare, aaPanel, LiteLLM, SearXNG,
  Wayback, A-Parser, PostgreSQL, РКН/antizapret, Alembic, git — это продукты/протоколы,
  не англицизмы. Названия статусов и действий — да; имена внешних систем — нет.

**Тесты:** `test_labels.py` — каждый ключ каждого enum (перечислить статусы из
`domain.py` и reject-коды из воронки) имеет перевод; неизвестный ключ → возвращается сам;
None → пустая строка или сырое (выбрать: **None → ""**). Фильтр зарегистрирован (рендер
шаблона со статусом даёт русское слово).

**Файлы:** создать `services/labels.py`, `tests/test_labels.py`; правки `app/api/panel.py`
(регистрация фильтров на объекте `templates`), `templates/{domains,queue,site,dashboard}.html`
(замена литералов на фильтры), `templates/domains.html` (легенда — русские фразы + коды).

## §B. Визуальный прогрессбар (переиспользуемый компонент)

**Проблема:** длинные Discovery/Score показывают текст `score: 5/5 — собираю: X`.
Пользователь просил **реальный визуальный бар**.

**Решение:** компонент в `base.html` (CSS-контракт) + апгрейд поллинг-JS. Механику
(`services/jobs.py`, `/run/{job}/progress` → `{running, done, total, current, error}`,
терминальный контракт) **не меняем** — только представление.

- **CSS** (`base.html`): класс `.progress` — трек (`--panel2`/`--line`) + заливка
  (`--amber`), подпись слева (имя джоба + `current`), процент справа (`--mono`). Высота
  ~18px, скруглён. Состояния: `.progress.done` заливка `--ok`; `.progress.err` заливка
  `--bad` + текст ошибки; **неопределённый** режим (`total==0` и `running`) — полосатая
  анимированная заливка (CSS `@keyframes`, уважает `prefers-reduced-motion`).
- **Разметка:** `#prog` из `hint`-строки превращается в контейнер бара (скрыт при
  простое, показывается когда джоб идёт/упал/только завершился).
- **JS:** апгрейд функции `poll(job)` в `domains.html` — вместо `el.textContent=…` пишет
  ширину заливки `= done/total*100%`, подпись, процент. **Терминальный контракт держим
  байт-в-байт:** `error` проверяется ПЕРВЫМ (класс `err`, без reload — оператор видит),
  `running==false` без ошибки = `done` (класс `done`, затем `location.reload()`),
  `total==0 && running` = неопределённый режим. Имя джоба в подписи — локализованное
  (`Поиск дропов`/`Запуск проверки`), маппинг в JS-объекте.
- **Переиспользуемость:** компонент самодостаточен (класс + JS-функция принимает
  `job`-имя и id контейнера). Подключаем на /domains для Discovery/Score. Форма готова
  под будущие длинные задачи M3–M5 — **но их сейчас НЕ подключаем** (нет джобов — YAGNI).

**Тесты:** JS без headless-фреймворка не юнит-тестируем — вместо этого проверяем
**серверный контракт** (уже покрыт `test_jobs.py`: `/run/{job}/progress` shape).
Визуальную проверку бара делаем **глазами** (локальный serve + Playwright-скриншот
идущего и завершённого джоба — как в Спек 1). Отметить в плане как ручной шаг.

**Файлы:** `base.html` (CSS `.progress`), `templates/domains.html` (разметка `#prog` +
JS `poll`).

## §C. Умный дашборд /diag

**Проблема:** /diag — плоская таблица пингов. Пользователь просил **дашборд
зависимостей**. Реальный `docker ps` отклонён (панель внутри контейнера, сокет не
проброшен — не расширяем attack surface LAN-бокса). Значит: обогащаем пинги, не
контейнеры.

**Решение:** +2 поля в спеке проверки, перегруппировка шаблона, сводка здоровья.

- **`_spec()`** (`diagnostics.py`): расширить кортеж `(key, label, role, need_cred, fn)`
  → `(key, label, role, need_cred, module, critical, fn)`. `module` ∈
  {`M1`,`M3`,`M4`,`M5`,`инфра`}; `critical` ∈ {True,False}.
  - Критичные (`critical=True`): **A-Parser** (вся воронка + витрины), **LiteLLM**
    (контент), **PostgreSQL** (БД), **Backorder** (discovery), **Wayback**+**РКН** (гейты
    M1). Опциональные (`critical=False`): **Cloudflare**, **aaPanel** (нужны только на
    провижне M3, кред может быть не задан), **SearXNG** (эхо/конкуренты — деградируемо).
  - `_run_one` и `run_diagnostics` — пробросить новые поля в результат-дикт (`module`,
    `critical` в возвращаемом словаре).
- **Сводка здоровья** (шапка /diag): вместо «N / M на связи» — «**N / M на связи ·
  критичные: все ✓**» или «**критичные: A-Parser, БД ✗**» (красным). Считается на
  сервере (роут `diag_view` в `panel.py`): `crit_down = [c for c in checks if c.critical
  and c.status=='fail']`.
- **Группировка** (diag.html): проверки сгруппированы по `module` (подзаголовки «M1 —
  поиск и скоринг», «M3 — провижн», «M4 — контент», «M5 — публикация», «Инфраструктура»).
  Внутри группы — та же строка, но:
  - **критичный упал** = красная тревога (жирный `led-bad` + плашка); **опц. упал** =
    приглушённо (обычный `led-off`/`led-bad`, без тревоги).
  - **отклик** — мелкий бар (доля от `PING_TIMEOUT`, класс из §B или простой inline),
    цвет: быстрый зелёный / медленный (>50% таймаута) янтарный.
- Пинг-при-открытии + ручной refresh + блок версии/git-pull — **как есть**, не трогаем.

**Тесты:** `test_diagnostics.py` (или в `test_web_fixes.py`) — каждый элемент `_spec()`
имеет `module` и `critical`; `run_diagnostics()` прокидывает их в результат; `diag_view`
считает `crit_down` корректно (мок: критичный fail → в списке, опц. fail → нет).

**Файлы:** `services/diagnostics.py` (кортеж + прокидка), `api/panel.py` (`diag_view` —
сводка `crit_down`), `templates/diag.html` (группировка + сводка + бар отклика).

## §D. Мелкий полиш (закрыть косметические долги Спек 1)

Три пункта из леджера Спек 1, каждый — точечная правка в соседнем файле:

1. **`price_checked_at` в тултипе домена** (`domains.html`): в тултипе бейджа-источника,
   где уже `· цена {N}`, добавить `· обновлено {{ d.price_checked_at.strftime('%d.%m') }}`
   (если поле не пусто). Данные уже пишутся (Спек 1 Task 5), просто не показывались.
2. **single-score «discovered (None)»** (`panel.py`, роут одиночного скоринга домена):
   когда `score_domain` вернул `unresolved` (whois не определил → домен остался
   `discovered`), флеш не должен показывать `→ discovered (None)`. Показать:
   «**whois не определил — домен остался в поиске, перепроверю на следующем прогоне**».
   Ветка: `if out.get('unresolved'): msg = "…"` до формата `→ {status} ({score})`.
3. **`period[0].get("id")` хардненинг** (`integrations/backorder.py`, `get_tariffs`):
   заменить bracket-доступ `period[0]["id"]` на `.get("id")` (живой тариф без `id` в
   period → не `KeyError`, а мягкий None; `period_id` ниже по коду не критичен).

**Тесты:** полиш-1 — рендер тултипа с `price_checked_at` (в `test_web_fixes.py`); полиш-2
— роут одиночного скоринга на unresolved-домене даёт русский флеш, не «(None)»; полиш-3 —
`get_tariffs` на тарифе без `period[].id` не падает (в `test_pricing.py`).

## §E. Границы (что НЕ входит)

- **Автономия:** тумблеры руками/автоматом по стадиям, шедулер, swappable-редактурный
  гейт, уведомления — **Спек 3 «Комбайн»**. Здесь только представление.
- **Реальный docker ps / контроль контейнеров** — отклонено (нет сокета, security).
- **Live-поллинг /diag** (авто-обновление без перезагрузки) — YAGNI; ручной refresh есть.
- **Перевод собственных имён сервисов** — намеренно нет (продукты, не англицизмы).
- **Новые длинные джобы M3–M5** под прогрессбар — компонент готов, подключение позже.
- Хард-гейты (деньги `confirmed_by_human`, публикация из `edited`) — не касаемся.

## §F. Инварианты и контракт

- **Дизайн:** светлая CMS, принцип шильдика, только CSS-переменные `base.html`
  (`--amber`/`--amber-soft`/`--amber2`/`--ok`/`--bad`/`--mut`/`--panel2`/`--line`/`--mono`
  …). Никакого тёмного/индустриального. Новые контролы/подписи — сразу по-русски.
- **Терминальный контракт прогресса** (`error`-first, `running==false`=готово,
  `total==0`=неопределённый) — держать байт-в-байт при апгрейде JS.
- **Безопасность панели:** CSRF same-origin + Basic-auth не ослаблять; правки — в тех же
  роутерах.
- **Тесты офлайн-детерминированы** (SQLite-харнесс, сеть мокается). Прогон:
  `.venv/bin/python -m pytest backend/tests/ -q`; pyflakes чистый.
- **Локализация — одна точка правды** (`labels.py`), не разбросанные литералы.

## §G. Фазы (для плана)

Порядок дёшево→ценно, каждая фаза — отдельно тестируема:

1. **labels.py + фильтры** — мапы, регистрация, тесты покрытия (фундамент §A).
2. **Локализация шаблонов** — заменить литералы статусов/лейнов/reject/действий на
   фильтры и русские подписи (§A применение).
3. **Прогрессбар** — CSS-компонент + апгрейд JS, ручная визуальная проверка (§B).
4. **/diag дашборд** — кортеж +module/critical, сводка crit_down, группировка шаблона (§C).
5. **Полиш** — три точечные правки + их тесты (§D).
