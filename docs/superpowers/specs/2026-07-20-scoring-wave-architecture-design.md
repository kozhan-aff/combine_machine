# Волновая архитектура воронки скоринга (M1) — дизайн

## Контекст

Сегодня `scoring.score_pending()` гоняет домены СТРОГО последовательно: домен A целиком
проходит `_funnel()` (T0 RD → T1 whois → T2 РКН/блэклист/SafeBrowsing/эхо → T3 Wayback →
T3b Ahrefs) с ранним выходом, ПОТОМ начинается домен B. Порядок стадий дёшево→дорого и
ранний выход уже на месте и подтверждены живыми данными — отклонённые на T0/T1 домены
выходят почти мгновенно, это НЕ баг. Но у оператора нет иного взгляда на прогон, кроме
«что происходит с ТЕКУЩИМ одним доменом прямо сейчас» (чипы стадий в `jobCard()`,
`base.html:494-505`) — а вопрос оператора другой: «как воронка отсеивает ВЕСЬ пул —
сколько отвалилось на whois, сколько на РКН, сколько дошло до дорогого Wayback».

Уже отгружено в этой же сессии (коммит `a73b85d`) — живая раскладка исхода прогона
(`_funnel_tally()`, `panel.py:676-720`): сколько домена решено «дёшево» (до Wayback) и
сколько дошло до дорогого T3, по данным `domain_score_log`. Это закрывает часть жалобы, но
НЕ меняет физику самого прогона: домены по-прежнему обрабатываются один за другим, без
конкурентности — а whois/РКН/Wayback это I/O-bound вызовы, которые ждут сеть, а не CPU.

**Задача этого дизайна:** перестроить `score_pending()` в НАСТОЯЩИЕ волны — один
инструмент (например whois) прогоняется на весь батч сразу, отсеивает часть, следующий
инструмент (РКН/блэклист/SafeBrowsing/эхо) — на оставшемся меньшем пуле, и так далее по
возрастанию цены (whois → risk → Wayback-история → Ahrefs). Внутри волны — конкурентность
через `ThreadPoolExecutor` (httpx-клиенты синхронные, потокобезопасны для параллельного
использования; переход на asyncio означал бы переписать все интеграционные клиенты — вне
объёма). Цель по решению пользователя (AskUserQuestion): и заметно быстрее, и видимость
волн — оба сразу, без жёсткого SLA по времени. Границы конкурентности — хардкод-константы
(не `/settings`): «если домен занят whois — скипаем в этой волне, если в бане РКН —
скипаем» — то есть сами лимиты волн фиксированы в коде, а не крутятся оператором.

## Рассмотренные подходы

1. **Ограниченная конкурентность без перестройки** (мой изначальный вариант) — обернуть
   СЕГОДНЯШНИЙ `_funnel()` (полный путь одного домена) в `ThreadPoolExecutor`, гоняя N
   доменов параллельно, каждый — весь свой путь целиком. Дёшево в коде (один пул на всю
   воронку), даёт speedup, но НЕ даёт волновой видимости: РКН для домена A мог бы идти
   одновременно с Wayback для домена B — с точки зрения оператора это неотличимо от
   сегодняшнего «всё сразу на один домен», просто на N доменов одновременно.
2. **Настоящие волны** (выбор пользователя) — расщепить `_funnel()` на функции-стадии,
   гонять КАЖДУЮ стадию на всём выжившем пуле, прежде чем начать следующую. Дороже в коде
   (нужна модель состояния, переживающая несколько вызовов), но именно это отвечает на
   вопрос оператора «как волна проходит» и соответствует его личному ментальному пониманию
   стоимости стадий (сначала дешёвый whois, потом РКН на меньшем пуле и т.д.).
3. Полный асинхронный переезд (asyncio + httpx.AsyncClient везде) — отклонён: требует
   переписать `integrations/*` целиком, непропорционально объёму задачи.

**Выбрано: подход 2.**

## Архитектура

### 1. Модель состояния — `FunnelState`

Домен на пути через волны — не ORM-объект, а лёгкий снэпшот (без привязки к сессии
SQLAlchemy, чтобы можно было держать в памяти между волнами без открытой транзакции):

```python
@dataclass
class FunnelState:
    domain_id: int
    domain: str
    lane: str | None
    referring_domains: int | None
    acquire_deadline: datetime | None
    sig: dict = field(default_factory=lambda: {"errors": []})
    reject_reason: str | None = None
    unresolved_why: str | None = None
    alive: bool = True
```

`score_pending()` строит список `states` ОДНИМ SQL-запросом (расширение сегодняшнего —
дозагрузить `lane`, `referring_domains`, `acquire_deadline` в SELECT рядом с `id`/`domain`,
запрос и сортировка по срочности не меняются).

### 2. Стадии-функции + конкурентность + вежливость

Константы конкурентности — хардкод в `scoring.py`, НЕ `/settings` (решение пользователя):

```python
_CONCURRENCY = {"whois": 12, "risk": 12, "history": 4, "ahrefs": 2}
```

`history=4` — жёсткий, неконфигурируемый потолок вежливости к archive.org (проектная
ценность, см. CLAUDE.md), `ahrefs=2` — капча за каждый вызов, дорого и хрупко к нагрузке.

Волны (в порядке прогона), каждая — чистая пересадка соответствующего куска сегодняшнего
`_funnel()` на `ThreadPoolExecutor`:

- **`_wave_t0(states)`** — RD/feed_flags фильтр, без сети, мгновенно, без пула.
- **`_wave_whois(states, clients, budget, run)`** — пул `_CONCURRENCY["whois"]`. На домен:
  бюджет (`budget.take()`), whois-роутер (TCI/A-Parser, уже с предохранителями), лейн-
  логика (bid/free/waiting/taken/unknown), проекция дедлайна, гейт возраста. Реджекты:
  `too_young`, `not_acquirable`; unresolved: `budget`/`whois_failed`/`waiting`/
  `whois_unclear`/`taken_undated`.
- **`_wave_risk(states, clients, run)`** — пул `_CONCURRENCY["risk"]`. На домен: РКН →
  блэклист → SafeBrowsing (с уже существующим предохранителем A-Parser) → indexed_echo
  (SearXNG). Эхо НЕ отбраковывает (сигнал score, не гейт) — но это тот же самый сетевой
  поход, поэтому живёт в этой же волне/пуле, не отдельной. Реджекты: `rkn`, `blacklist`,
  `safebrowsing`.
- **`_wave_history(states, clients, run)`** — пул `_CONCURRENCY["history"]`. Wayback
  `classify_history`, категорийный hard-reject, фолбэк возраста по первому снимку.
  Реджект: `history_dirty`.
- **`_wave_ahrefs(states, clients, budget, run)`** — пул `_CONCURRENCY["ahrefs"]`. Только
  для доменов без RD из фида (см. сегодняшнее условие), под бюджетом. Не отбраковывает.

Оркестратор:

```python
def _run_waves(states, clients, st, run) -> None:
    _wave_t0(states)
    _checkpoint(states, run, "rd")
    _wave_whois(states, clients, Budget(st["max_whois_per_run"]), run)
    _checkpoint(states, run, "whois")
    _wave_risk(states, clients, run)
    _checkpoint(states, run, "risk")
    _wave_history(states, clients, run)
    _checkpoint(states, run, "history")
    _wave_ahrefs(states, clients, Budget(st["max_ahrefs_per_run"]), run)
```

`_checkpoint` фиксирует границу волны: домены с `reject_reason` или `unresolved_why` этой
волны финализируются в БД немедленно (см. §4) и помечаются `alive=False`; выжившие идут
в следующую волну как уменьшившийся список. Это и есть видимость «отсеялись НА ЭТОЙ
стадии», а не «когда-то потом».

### 3. Бюджеты и отмена под конкурентностью

Сегодняшний бюджет — мутируемый `[int]`, не потокобезопасный (нормально при
последовательной обработке, ломается при параллельной). Замена:

```python
class Budget:
    def __init__(self, n: int):
        self._n = n
        self._lock = threading.Lock()
    def take(self) -> bool:
        with self._lock:
            if self._n <= 0:
                return False
            self._n -= 1
            return True
```

Порядок расходования бюджета внутри волны становится приблизительным (не строго
порядковым, как сегодня) — домены в пуле из 12 воркеров завершаются не в порядке
отправки. Батч уже отсортирован по срочности ДО волн (без изменений), поэтому порядок
расходования остаётся близким к приоритетному, но не строго гарантированным впритык к
границе бюджета. Это документированное, приемлемое изменение поведения — не регрессия
инварианта (бюджет всё ещё расходуется РОВНО N раз, не больше).

Отмена (кнопка «✕ Отменить», `jobs.cancelled(run)`):
- **Между волнами** — всегда проверяется перед стартом следующей волны.
- **Внутри волны** — воркеры отправляются в пул, результаты собираются через
  `concurrent.futures.as_completed()`; после каждого завершённого результата проверяется
  флаг отмены. При срабатывании: `executor.shutdown(wait=False, cancel_futures=True)`
  (доступно в Python 3.9+, стек проекта — 3.12), уже завершённые к этому моменту домены
  финализированы (их DB-запись уже произошла в момент завершения — см. §4), незавершённые
  остаются `discovered` (как и сегодня — частичный прогресс не теряется), поднимается
  `jobs.Cancelled()`.

### 4. Финализация — запись в БД

Сегодняшний `score_domain()` после `_funnel()` пишет решение (статус/score/breakdown/
`DomainScoreLog`) в ОДНОЙ функции. В волновой модели запись должна происходить в МОМЕНТ,
когда `FunnelState` выходит из конвейера — будь то реджект/unresolved на границе волны,
либо финальное решение после последней волны:

```python
def _commit_result(state: FunnelState, run: int, st: dict) -> None:
    with SessionLocal() as db:
        d = db.get(Domain, state.domain_id)
        if d is None or d.status not in ("discovered", "scored", "rejected"):
            return   # статус увели параллельным процессом/оператором — не наш домен больше
        ...  # тот же код, что сегодня в score_domain() ПОСЛЕ вызова _funnel (строки 684-811):
             # unresolved-ветка / reject-ветка / compute_score-ветка, один DomainScoreLog
```

Это прямой перенос существующего хвоста `score_domain()` (после строки, где сегодня
вызывается `_funnel`) в отдельную функцию, принимающую `FunnelState` вместо доменного ORM-
объекта и `sig`. Каждый вызов открывает СВОЮ сессию — тот же паттерн, что и сегодня
(домены на общий session не претендуют, конкурентная запись безопасна: разные `domain_id`
— разные строки).

### 5. `score_domain()` — контракт не меняется

Внешняя сигнатура и поведение `score_domain(domain_id, clients=None, whois_budget=None,
ahrefs_budget=None, run=None) -> dict` остаются как есть — это гарантирует, что ВЕСЬ
сегодняшний `test_funnel.py` (~45 тестов, единственный, кто дёргает `score_domain`/
`_funnel` напрямую) остаётся зелёным БЕЗ ИЗМЕНЕНИЙ. Внутри `score_domain()` строит
одиночный `FunnelState`, прогоняет его через `_run_waves([state], ...)` (батч размера 1 —
волны линеаризуются в тот же порядок стадий, что и сегодняшний `_funnel`) и коммитит
результат. `_funnel()` как отдельная функция УДАЛЯЕТСЯ (логика расходится по волновым
функциям) — тесты, которые сегодня бьют напрямую по `_funnel()` (если такие есть отдельно
от `score_domain()`), потребуют точечной правки на вызов через `score_domain()`/новые
волновые хелперы; это выясняется на этапе плана чтением `test_funnel.py` целиком.

### 6. UI — прогрессбар и волновая история

Пользователь подтвердил направление и отдельно указал: **сам блок прогрессбара можно
увеличить** — сегодняшний `.job` (`base.html:328-351`) компактный (одна строка чипов +
одна строка tally + одна полоса), волновая раскладка получает больше места без ужимания
в существующие узкие блоки.

- `job_run.stage` = ключ ТЕКУЩЕЙ ВОЛНЫ (`rd`/`whois`/`risk`/`history`/`ahrefs` — эхо
  сливается в `risk`, отдельного чипа/стадии для него больше нет, см. §2). `_advance()`
  (`jobs.py:282-298`) перестаёт быть «пересчитать заново на каждом домене» — с волнами
  стадия РЕАЛЬНО пройдена для всего батча, прежде чем начинается следующая, поэтому чипы
  становятся монотонными (before → done, once) без специальной оговорки в докстринге.
- `done`/`total` — прогресс ВНУТРИ текущей волны (сколько из вошедших в неё доменов уже
  обработано), не «домены прогона всего» — иначе прогресс-бар откатывался бы назад при
  входе в новую (меньшую) волну.
- Волновая история — новый блок под чипами, крупнее сегодняшнего `.job-tally`: список
  «RD: 5510 → 4310 выжило · whois: 4310 → 3800 · risk: 3800 → 3100 · history: идёт,
  1200/3100» — по одной строке на КАЖДУЮ волну, пройденную в этом прогоне (не только
  текущую). Новой колонки/миграции не нужно: `_checkpoint` (см. §2) после каждой волны
  перестраивает всю waterfall-строку целиком (счётчики выживших он и так знает —
  `len(states)` до/после фильтра в памяти оркестратора) и пишет её в уже существующее поле
  `job_run.message` (`String(400)`, `jobs.py:35`) через `jobs.report(run, message=...)` —
  тот же канал, что сегодня несёт финальную сводку прогона. Формат укладывается в лимит
  (5 волн × ~30 символов ≈ 150 символов). Панель просто показывает `j.message` как есть,
  без нового парсинга на фронте.
- Существующий `_funnel_tally()` (по outcome/reject_reason) остаётся — он уже показывает
  разбивку ПО ПРИЧИНЕ отказа для всего прогона; волновая история — ДОПОЛНЯЮЩИЙ, не
  заменяющий блок (разная гранулярность: причина vs. стадия).

## Тестирование

- **`test_funnel.py` остаётся зелёным БЕЗ ИЗМЕНЕНИЙ** — главный регрессионный барьер:
  весь сегодняшний контракт `score_domain()` (лейны, гейты возраста, предохранители,
  whitelist-запись колонок, `_decide()`-гарды) обязан пройти как есть, раз внешнее
  поведение не меняется.
- Новые тесты нужны для:
  - оркестратора волн (счётчик выживших уменьшается корректно между волнами;
    `_checkpoint` финализирует ровно тех, у кого выставлен `reject_reason`/
    `unresolved_why`, и не трогает выживших);
  - `Budget` под конкурентной нагрузкой (N потоков, ровно N `take()` успешны при бюджете
    N, остальные — False; стресс через `ThreadPoolExecutor` с большим N);
  - отмены между волнами и ВНУТРИ волны (частичный `done` сохраняется, `Cancelled`
    поднимается, уже завершённые домены зафиксированы в БД);
  - сжатия пула по волнам (100 доменов → 40 после whois → 10 после risk — санитарный тест
    на реальных фейковых клиентах, без сети).
- **Обязательный последующий шаг** (не блокирует мерж, но без него дизайн не подтверждён
  вживую): живой прогон на боксе (`192.168.1.77`) — проверить реальный выигрыш по времени
  и реальное поведение предохранителей/бюджетов под конкурентностью A-Parser/TCI, которых
  оффлайн-тесты не воспроизводят.

## Не входит в объём этого дизайна

- `/settings`-контролы для констант конкурентности — сознательно исключены по решению
  пользователя (хардкод, не рантайм-настройка).
- Переезд на asyncio/`httpx.AsyncClient` — вне объёма, `integrations/` остаются
  синхронными.
- Изменение самих правил гейтов/скоринга (пороги, веса, hard-reject флаги) — не
  затрагивается, только физика прогона.
