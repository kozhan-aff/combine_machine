# История решений скоринга по домену — дизайн

**Запрос:** "прямая система логирования, чтобы Claude Code видел, что происходит в
машине". Уточнено диалогом: не про доступность (панель по HTTP уже доступна, когда
бокс жив — `job_run`/`/api/jobs/live`/`/diag`), а про ГЛУБИНУ данных. `job_run`
показывает только текущую стадию активной задачи; `Domain.score_breakdown` — один
JSONB, перезаписывается на каждом рескоре, история решений теряется безвозвратно.
Приоритет — решения скоринга по конкретному домену (T0-T3 воронка): что происходило
на КАЖДОМ прогоне, не только на последнем.

## Решение (Вариант А, согласован в диалоге)

Новая append-only таблица `domain_score_log` — расширение уже работающего паттерна
`job_run` (кросс-процессный реестр в PostgreSQL), а не новая инфраструктура (отвергнут
Вариант Б — структурные логи + Loki/аналог: новый контейнер на боксе ради задачи,
которая сейчас касается конкретно скоринга, нарушает ladder-принцип «уже работающее
решает — используй его»).

## Точка записи

**Единственное место — `scoring.score_domain()`, не `_funnel()`.** `score_domain()` —
уже существующая функция (строки ~556-676 `scoring.py`), которая гоняет `_funnel()` и
ПЕРСИСТИТ результат в `Domain` на каждом из трёх путей возврата:
1. `acquirability_unresolved` (whois не разобрал/бюджет исчерпан — домен остаётся
   `discovered`, строки ~579-597).
2. `reject` (воронка отклонила — строка ~599-600).
3. Полный `compute_score` + `_decide` (домен дошёл до конца воронки — строки ~601-676).

Все три пути уже имеют под рукой `sig` (полный словарь сигналов), `d` (домен), `run`
(id job_run, уже прокидывается параметром) и — на путях 2/3 — `result`/`reject`. Задача
— добавить ОДНУ строку записи в `domain_score_log` перед каждым `return`, без
дублирования уже посчитанного. Никакой новой логики скоринга не добавляется — это
чистый side-effect персистентности поверх существующих решений.

## Схема

Миграция `0024_domain_score_log`:

```python
domain_id: int (FK domains.id, NOT NULL)
run_id: int | None (FK job_run.id, NULL — есть вызовы score_domain() вне трекнутого
                     job, например ручной "▶" на одном домене из /domains)
outcome: str  # 'unresolved' | 'rejected' | 'scored'
reject_reason: str | None       # зеркалит Domain.reject_reason на момент ЭТОГО прогона
score: float | None             # зеркалит Domain.score на момент ЭТОГО прогона
sig: JSONB                      # полный снимок sig из _funnel() — то, что сейчас теряется
created_at: DateTime(timezone=True), server_default=now()
```

Индекс `(domain_id, created_at)` — типичный запрос: «история этого домена, новые
сверху».

## Точка чтения

`GET /api/domains/{domain_id}/score-history?limit=50` (панель, тот же файл/паттерн,
что `GET /api/jobs/live`) — JSON-список строк, `created_at DESC`, лимит по умолчанию
50 (query-параметром можно запросить больше). Простой `SELECT ... ORDER BY
created_at DESC LIMIT :limit` — ничего сложнее.

## Хард-гейты

Не затронуты вообще — таблица чисто наблюдательная (append-only лог решений
скоринга), ни денежный, ни редактурный гейт этот код не видит и не может видеть
(никаких вызовов `confirm_order`/`execute_confirmed_order`/`mark_caught`/`mark_edited`).

## Что НЕ делаем

- Не логируем `recheck_acquirability()` в эту же таблицу в этой итерации — она не
  зовёт `_funnel()`/`score_domain()` целиком (более лёгкий путь, только
  `acquirability_verdict()`), другая форма данных. Если понадобится — отдельное
  расширение, не в этой задаче (YAGNI: нет запроса на историю перепроверок занятости,
  только на историю СКОРИНГА).
- Не добавляем retention/TTL — рескор происходит не чаще раза в сутки на домен
  (`RECHECK_EVERY`), объём данных небольшой; чистка — отдельная задача, если/когда
  реально понадобится (не гадаем объём заранее).
- Не меняем `Domain.score_breakdown` — она остаётся «последний снимок» для UI-бейджей
  (карточка домена и так уже её использует), `domain_score_log` — история РЯДОМ, не
  замена.

## Тестирование

- `score_domain()` пишет ровно одну строку `domain_score_log` на каждом из трёх путей
  (unresolved/rejected/scored) — три теста, каждый мокает `_funnel()` на нужную ветку,
  проверяет содержимое новой строки.
- Повторный вызов `score_domain()` на том же домене добавляет ВТОРУЮ строку, не
  перезаписывает первую (это и есть смысл фичи — прямой регресс-тест на историчность).
- Эндпоинт `/api/domains/{id}/score-history` — отдаёт строки в правильном порядке
  (новые сверху), уважает `limit`, пустой список для домена без истории (не 404).

## Самопроверка

- Плейсхолдеров нет.
- Внутренних противоречий нет — единая точка записи, единая точка чтения.
- Масштаб — один связный кусок (append-only таблица + запись в одном месте + один
  read-эндпоинт), одна задача в плане, декомпозиции не требует.
- Неоднозначность: «нужен ли `run_id` NOT NULL» — разрешена явно (nullable, ручной
  вызов на одном домене вне job_run — легитимный кейс, `/domains/{domain_id}/score`
  роут это уже делает, см. `panel.py:674-676`).
