# Резервный оффер при публикации — дизайн

## Контекст

Аудит 2026-07-15 (F3) нашёл: `p.offer_id` фиксируется НАВСЕГДА в момент генерации страницы
(F26, миграция 0018) и публикация его не переоценивает — если оффер выключили
(`Offer.active=False`) после генерации, страница публикуется со ссылкой на уже выключенный
оффер как есть.

Пользователь принял решение (2026-07-16): публикацию **не блокировать** — `offer_id` это
зафиксированное решение о бренде/контенте, трогать его при публикации нельзя (иначе страница
про бренд A вдруг сошлётся на бренд B — ровно тот баг, который чинил F26). Вместо блокировки —
две меры:

1. **Видимость** — бейдж «оффер выключен» на карточке сайта. **Уже сделано и смержено**
   (коммит `5750bc5`, main).
2. **Резервный URL** — если оффер выключен, ссылка на странице ведёт не в никуда, а на общий
   резервный URL (сводная страница сравнения VPN / собственный лендинг). Этот документ —
   дизайн пункта 2.

## Решения пользователя (зафиксированы в диалоге)

| Вопрос | Решение |
|---|---|
| Триггер подмены | ТОЛЬКО `Offer.active == False`. Не проверка живости ссылки (HTTP-запрос) — это отдельная, более сложная фича, не в этом дизайне. |
| Область резерва | Один глобальный URL на весь портфель. Не по гео/языку, не per-offer. |
| Что меняется в тексте | ТОЛЬКО ссылка (`href`). Текст статьи, бренд в кнопке («Перейти к NordVPN»), промокод — остаются как сгенерированы. Кнопка может называть бренд, у которого уже нет резерва за ней — осознанный компромисс простоты. |
| Когда применяется | ТОЛЬКО при публикации ещё НЕ опубликованных страниц (`edited -> published`). Уже опубликованные страницы с мёртвой ссылкой резерв не чинит — это ручное решение оператора (republish), сигнал уже даёт бейдж. |

## Архитектура

**Хранение:** новая single-row таблица `OfferSettings` (id=1, `reserve_offer_url: str | None`) —
тот же паттерн, что уже используют `ScoringSettings`/`AutonomySettings` (single-row рантайм-конфиг,
редактируемый через UI, дефолт NULL = резерв не настроен → сегодняшнее поведение). Живёт в
`backend/app/models/offer.py` рядом с `Offer`/`SiteOffer` (это оффер-специфичная настройка, не
скоринг и не автономия).

**Точка подстановки:** `backend/app/services/content.py::render_html()` — та же функция, что уже
несёт F28 (не рендерить опасную схему `href`). Резервная логика встаёт РЯДОМ с этой проверкой, не
отдельным местом:

```python
def render_html(page, offer=None, lang: str = "ru", reserve_url: str | None = None) -> str:
    ...
    link = offer.affiliate_link if offer is not None else None
    if offer is not None and not offer.active and reserve_url:
        link = reserve_url        # F3: выключенный оффер -> резерв, если он настроен
    if offer is not None and link and is_safe_url(link):
        ...рендерим блок с offer.brand/offer.promo_code как раньше, href=link...
```

`reserve_url` — новый keyword-параметр с дефолтом `None` (обратная совместимость: единственный
продакшн-вызов — `publish.py:121`; self-test функции внизу `content.py` не передаёт его и не
должен сломаться).

**Кто читает настройку:** `publish.py::publish_site()` уже открывает `db`-сессию и уже резолвит
`fallback_offer` один раз на весь вызов (F26-комментарий) — резерв читается там же, один раз на
`publish_site()`, а не на страницу (одна и та же настройка для всех страниц одной публикации).

**Почему не трогаем `_pick_offer`/legacy-путь:** легаси-страницы без `p.offer_id` (созданные до
0018) уже используют `fallback_offer`, который САМ фильтрует `active=True` — для них
выключенный оффер и так никогда не попадёт на страницу. Резервная логика нужна только ветке
`offer = db.get(Offer, p.offer_id)` (страницы с зафиксированным оффером).

## UI

Малая форма на `/offers` (это экран конфигурации офферов, не `/settings` — резерв про офферы,
не про скоринг доноров): одно поле `reserve_offer_url` + кнопка сохранить, тот же визуальный
паттерн, что форма создания оффера на этом же экране. Валидация — тот же `is_safe_url()`, что уже
используют форма создания оффера и `render_html` (http/https, без `javascript:`/`data:`).

Бейдж «оффер выключен» (уже смержен, `site.html`) получает уточнение в `title`: если резерв
настроен — «при следующей публикации поведёт на резервный URL»; если нет — «ссылка мертва, резерв
не настроен» (эксплуатирует ту же переменную, но это чисто текстовая правка `title`, не меняет
условие показа бейджа).

## Миграция

Новая линейная миграция (голова на момент этого спека — `0021_site_last_attempt`, следующая —
`0022`), создающая `offer_settings` с единственной колонкой `reserve_offer_url TEXT NULL` и
сидом строки `id=1` со значением `NULL` (как это уже делает миграция `AutonomySettings`).

## Что осознанно НЕ входит в этот дизайн

- Проверка реальной живости ссылки (HTTP HEAD/GET на `affiliate_link`) — отдельная, более сложная
  фича с вопросами таймаутов/капов/периодичности; пользователь пометил как «надо прорабатывать
  отдельно», в этот дизайн не включена.
- Ремонт уже опубликованных страниц с мёртвой ссылкой — ручной republish оператором, сигнал даёт
  существующий бейдж.
- Резерв per-offer/per-гео/per-язык — пользователь выбрал один глобальный URL.
- Изменение брендового текста/промокода в блоке оффера при резерве — пользователь выбрал менять
  только `href`.

## Критерии приёмки

- Оффер активен → страница публикуется с его реальной ссылкой (поведение не меняется).
- Оффер выключен, резерв НЕ настроен (`reserve_offer_url IS NULL`) → страница публикуется с
  прежней (уже неактивной) ссылкой оффера — сегодняшнее поведение, ничего не ломаем при пустой
  настройке.
- Оффер выключен, резерв настроен и безопасен (`is_safe_url`) → страница публикует ссылку на
  резервный URL, бренд/промокод в тексте не меняются.
- Резервный URL с опасной схемой (`javascript:`/`data:`) через форму `/offers` не сохраняется
  (валидация на входе, как у создания оффера) — и даже если каким-то путём попадёт в БД,
  `render_html` его не отрендерит (defense in depth, тот же принцип, что F28).
- Уже опубликованная страница с выключенным оффером НЕ трогается автоматически.
- Легаси-страницы (`p.offer_id IS NULL`) ведут себя как раньше — резервная логика их не касается.
