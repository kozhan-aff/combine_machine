# Наблюдаемость машины + M1 как инбокс решений — дизайн

**Дата:** 2026-07-12
**Ветка:** `feat/panel-m1-observability`
**Статус:** утверждён

## 1. Проблема

Три симптома, один корень — **машина не рассказывает, что делает**.

**1.1 Прогресс-бар перепроверки не работает вообще.** [panel.py:365](../../../backend/app/api/panel.py#L365)
пускает в `GET /run/{job}/progress` только `discovery|score|sweep`, а
[domains.html:138](../../../backend/app/templates/domains.html#L138) поллит ещё и `recheck` → 404 →
полоса молча не появляется. Оператор видит зелёный баннер «Перепроверка занятости запущена…» и
больше ничего: баннер соврал, что что-то происходит.

**1.2 Пульт физически не может видеть автопилот.** [jobs.py](../../../backend/app/services/jobs.py) —
это `dict` в памяти процесса **backend**. Автопилотный свип крутится в контейнере **worker** —
другой процесс, другая память. Панель узнаёт о работе воркера только постфактум, строкой в
`AutonomyRun`. Автопилот работает вслепую, поэтому его страшно включать — и это правильный страх.

**1.3 Стадий в контракте прогресса нет.** `jobs.report(done, total, current)` — счётчик и имя
домена. Воронка скоринга (T0 RD → T1 whois → T2 РКН/блэклист/эхо → T3 Wayback → T3b Ahrefs)
размечена внутри `_funnel()` комментариями, но наружу не выходит. Показать «на каком этапе что
делается» сегодня нечем.

**1.4 Экран M1 — дамп базы, а не рабочее место.** 5827 строк, отсортированных по score, из них
5605 «найден» (сырой фид, делать с ними нечего). 18 доменов, ждущих решения, утонули в этом дампе.

**1.5 Инбокс не отличает «спорный» от «непроверенного» — дыра в гейте курации.**
[scoring.py:38](../../../backend/app/services/scoring.py#L38) не даёт авто-одобрить домен без
успешной проверки истории (гард держит) — домен уходит в `scored`, то есть **в инбокс к человеку**.
Значит: Wayback лежит час → в инбокс сыпется всё подряд вперемешку с реально спорными, и выглядит
там ровно так же — «оценён, score 0.86». Человек одобряет, считая, что машина историю проверила.
Машина её не проверяла. Гейт курации вырождается в штамповку — ровно в том инварианте (домены за
чистую историю), ради которого гейт и существует. Данные есть: `score_breakdown.errors` хранит
`wayback:ConnectError`. Их никто не показывает.

**1.6 Запущенную задачу нечем остановить.** «Проверить весь пул» на 5605 доменов — это часы работы и
сожжённые квоты A-Parser; единственный способ прервать — рестарт контейнера.

## 2. Что делаем

Реестр задач переезжает в PostgreSQL и обрастает стадиями; Пульт становится диспетчерским экраном;
M1 из дампа превращается в очередь работы человека.

**Хард-гейты не трогаются.** Деньги (`confirmed_by_human=true`) и редактура (публикация только из
`edited`) остаются ровно там, где стоят. Пакетное одобрение из §6 — это по-прежнему решение
человека, просто одним кликом вместо восемнадцати; оркестратор к нему не прикасается.

## 3. Реестр задач в PostgreSQL

### 3.1 Модель

Новая таблица `job_run` (миграция `0007_job_run`):

| поле | тип | смысл |
|---|---|---|
| `id` | int PK | |
| `name` | str | `discovery` / `score` / `recheck` / `sweep` |
| `trigger` | str | `manual` (кнопка) / `auto` (воркер) |
| `status` | str | `running` / `done` / `failed` / `cancelled` |
| `stage` | str | ключ текущей стадии (`whois`, `cctld`, `provision`…) |
| `stages` | JSON | снимок степпера: `[{key, label, state}]`, state = `done`/`active`/`pending`/`skip` |
| `done`, `total` | int | счётчик прогресса; `total=0` = неопределённый режим |
| `current` | str | что сейчас в работе (домен / источник) |
| `message` | str | итог прогона («проверено 200: занято 3, свободны 197») |
| `error` | str \| None | текст падения |
| `cancel_requested` | bool | человек нажал «стоп» |
| `started_at`, `updated_at`, `finished_at` | datetime | |

Частичный уникальный индекс `(name) WHERE status='running'` — single-flight **между процессами**:
воркер не сможет запустить второй `score`, пока оператор гоняет свой (сегодня может, и они вдвоём
жгут квоту A-Parser). Индекс поддерживают и PostgreSQL, и SQLite (тестовый харнесс).

Свип свой замок уже имеет (`AutonomyRun._acquire_lock`, гонка там продумана) — его не дублируем:
`track("sweep", lock=False)`. `AutonomyRun` остаётся историей автопилота, `job_run` — прогрессом.

### 3.2 Контракт `services/jobs.py`

```python
jobs.track(name, *, trigger="manual", stages=None, lock=True)  # контекст-менеджер: строка в БД,
                                                               # ловит исключение → status=failed
jobs.spawn(name, target) -> bool       # фоновый поток (как сегодня start); False = уже идёт
jobs.report(name, done, total, current="", stage="", message="")
jobs.cancelled(name) -> bool           # сервис спрашивает между итерациями
jobs.request_cancel(name)
jobs.live() -> list[dict]              # все running — для Пульта и шапки
jobs.last(name) -> dict | None         # последний завершённый прогон (итог)
```

**Колбэки `on_progress` из сервисов убираются.** `run_discovery` / `score_pending` /
`recheck_acquirability` / `run_sweep` сами открывают `with jobs.track(...)` и репортят внутрь.
Кто бы их ни позвал — кнопка панели, оркестратор, cron воркера — прогресс появляется бесплатно.
Именно прокидывание колбэка только из панели и делало работу воркера невидимой.

`score_domain` (одиночный домен по кнопке в строке) записи реестра **не** открывает — он
мгновенный.

### 3.3 Терминальный контракт (сохраняется)

Проверять в порядке: `error != None` → упал; `status == 'running'` → идёт; иначе завершён успешно,
**даже если `done == total == 0`** (discovery без кандидатов). `done/total` — отображение, не признак
терминала. Крашнутый контейнер: строка `running` с `updated_at` старше 15 минут — показываем
«задача оборвалась» (переиспользуем порог `STALE_MIN` оркестратора).

## 4. Стадии — «на каком этапе что делается»

У каждой задачи свой степпер, отражающий её физику:

```
↻ Поиск дропов                                    ▓▓▓▓▓▓▒░░░
   backorder ✓4102 → cctld ✓1203 → reg.ru ◉ → sweb ○ → дедуп ○

▶ Проверка доменов                    12/50       ▓▓▓░░░░░░░
   сейчас: clara-c.ru
   RD ✓ → whois ✓ → РКН/блэклист ◉ → история ○ → Ahrefs ○      [■ стоп]

⟳ Перепроверка занятости              84/200      ▓▓▓▓░░░░░░
   whois: asmdivan.ru                                          [■ стоп]

✈ Свип автопилота (cron)              3/7         ▓▓▓▓░░░░░░
   поиск ✓ → скоринг ✓ → очередь ◉ → провижн ○ → контент ○ → публикация ○ → индексация ○
```

- **discovery**: стадии = включённые источники (из `sources_enabled`) + дедуп + запись. `total`
  заранее неизвестен → бар неопределённый, но степпер точный.
- **score**: общий бар = домены N/M; степпер = воронка **текущего** домена. `_funnel()` получает
  отметку стадии (`jobs.report(stage=...)`) в местах, где сегодня стоят комментарии T0–T3b.
- **recheck**: одна стадия (whois), бар N/M.
- **sweep**: стадии = `STAGES` оркестратора (7 ключей → русские лейблы); выключенные тумблером —
  `state=skip`, не исчезают, чтобы было видно, что стадия отключена, а не сломалась.

## 5. Пульт — диспетчерский экран

Сверху блок **«Машина сейчас»**: карточки всех активных задач с их степперами и кнопкой «■ стоп».
Автопилотный свип из воркера рисуется здесь наравне с ручными — тем же компонентом, потому что
данные лежат в одном месте.

Ничего не идёт — честное «машина простаивает» + итог последнего прогона каждой задачи («проверка:
200 доменов, 3 занято · 14:32»).

Тонкая строка того же прогресса — в шапке `base.html`, то есть на **любом** экране: ушёл в
Настройки, пока идёт проверка — всё равно видишь, что машина работает, и можешь остановить.

Транспорт — поллинг `GET /api/jobs/live` раз в 1.5 с (как сегодня). SSE не берём: лишняя сложность
ради одного оператора.

## 6. Экран M1 — четыре блока

**Блок 1 · Машина M1.** Три станции в ряд (поиск / проверка / перепроверка): что делает, когда
гонялась последний раз, итог, кнопка. Пока задача идёт, станция превращается в бар со степпером и
кнопкой «стоп». Шильдик (`<details>` «зачем это») сохраняется — дизайн-контракт.

**Блок 2 · Ждёт твоего решения** (`scored`). Компактная строка: домен, score-бар, RD, возраст,
сигналы, `[✓][✗]`. Плюс два новых элемента:

- **Пометка «оценён вслепую»** (§1.5). Если `score_breakdown.errors` содержит `wayback:` / `rkn:` /
  `blacklist:` / `searxng:` — строка помечается: «история не проверена: Wayback был недоступен».
  Рядом — «▶ перепроверить» (повторный `score_domain`). Человек больше не штампует непроверенное,
  думая, что оно проверено.
- **Сортировка по срочности.** Дефолт — `acquire_deadline` по возрастанию (nulls last), не по score.
  Домен, дропающийся завтра, стоит выше красивого с дропом через месяц. Над списком — «через 2 дня
  дроп: 3 домена». Переключатель «по score» остаётся.

**Пакетное решение.** Кнопка «одобрить все ≥ порога» с живым счётчиком «под раздачу попадёт N» и
явным исключением помеченных «вслепую» — их пакетом не одобряем, только руками (иначе пакет
превращается в обход гейта курации). Гейт остаётся человеческим: это по-прежнему клик человека.

**Блок 3 · Готовы к выкупу** (`approved`): домен, цена, дедлайн дропа, `[＋ в очередь выкупа]`,
`[🛒 купил руками]`. Мост в M2.

**Блок 4 · Воронка** — 5605 → 18 → 14 → 189, и по клику **разбор причин отказа**: сколько отсеяно по
`low_rd`, `feed_flag`, `too_young`, `rkn`, `blacklist`, `history_dirty`, `low_score`,
`not_acquirable`. Это обратная связь по порогам: видно, не режет ли порог лишнего.

**Полный пул** (5827 строк, фильтры, 11 колонок) уезжает на `/domains/pool` — он для расследований,
а не для ежедневной работы. Существующая таблица переносится туда целиком.

## 7. Баннеры

Баннеры `?msg=`/`?err=` **у запуска длинных задач** убираются. Нажал → сразу бар (JS рисует
«запускаю…» и начинает поллинг), а не зелёная плашка, которая врёт про «запущено» и живёт своей
жизнью. Итог прогона живёт в карточке задачи (`jobs.last`) и переживает перезагрузку страницы.

У мгновенных действий (одобрил домен, сохранил настройки, ошибка деплоя) короткий flash остаётся —
там показывать нечего. Баннер диагностики (`diag_alert`) — это состояние, а не уведомление, он
остаётся.

## 9. Тестирование (офлайн, герметично)

- `job_run`: single-flight — второй `track("score")` при живом первом падает/возвращает False;
  `cancel_requested` → сервис завершается со `status=cancelled` и частичным `done`.
- Терминальный контракт: `error` проверяется раньше `done/total`; `done==total==0` без ошибки = успех.
- Стадии: `score_pending` на моках проходит T0→T3b и репортит соответствующие `stage`;
  отсеянный на T1 домен НЕ доходит до Wayback (регрессия воронки сохраняется).
- Прогресс воркера: `run_sweep(trigger="cron")` без всяких колбэков создаёт `job_run` со `stages`
  из `STAGES` — то есть Пульт увидит автопилот.
- Инбокс: домен с `score_breakdown.errors=["wayback:ConnectError"]` помечен «вслепую» и **не
  попадает** в пакетное одобрение.
- Сортировка: домен с ближним `acquire_deadline` и меньшим score стоит выше.
- Разбор причин: счётчики по `reject_reason` сходятся с общим числом `rejected`.
- Хард-гейты: тест, что оркестратор по-прежнему не зовёт `confirm_order` / `mark_edited`.
- `pytest backend/tests/ -q` зелёный, pyflakes чист.

## 10. Что НЕ входит (следующие спеки)

- **Автопилот, которому можно доверять**: health-gate свипа (сегодня оркестратор не знает слова
  `diag` — лежащий Wayback не мешает ему прогнать 500 доменов вслепую), dry-run свип, красная
  кнопка «стоп всё».
- **Баланс-гард M2** (мелкий фикс, поедет попутно): не отправлять заказ при недостатке средств —
  сейчас он уходит при 0 ₽, виснет «не оплачен», домен не перехватывают.
- **Гейты в Telegram**: панель — pull-интерфейс; комбайну 24/7 нужен push с инлайн-кнопками ✓/✗.
- **Обратная связь скоринга**: связать признаки домена с исходом (за сколько дней проиндексировался).
  Без неё веса скоринга навсегда останутся догадкой.
- **Контент-план** вместо одного вызова LLM; **тематическое соответствие домена офферу** (инвариант
  «сайты тематически связные» в коде сегодня не охраняется ничем).
