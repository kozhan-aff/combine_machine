# Наблюдаемость машины + M1 как инбокс решений — дизайн

**Дата:** 2026-07-12
**Ветка:** `feat/panel-m1-observability`
**Статус:** утверждён

## 1. Проблема

Три симптома, один корень — **машина не рассказывает, что делает**.

**1.1 Прогресс-бар перепроверки не работает вообще.** [panel.py:365](../../../backend/app/api/panel.py#L365)
пускает в `GET /run/{job}/progress` только `discovery|score|sweep`, а
[domains.html:138](../../../backend/app/templates/domains.html#L138) поллит ещё и `recheck` → 404 →
полоса молча не появляется. Оператор видит зелёный баннер «Перепроверка занятости запущена…» и
больше ничего: баннер соврал, что что-то происходит.

**1.2 Пульт физически не может видеть автопилот.** [jobs.py](../../../backend/app/services/jobs.py) —
это `dict` в памяти процесса **backend**. Автопилотный свип крутится в контейнере **worker** —
другой процесс, другая память. Панель узнаёт о работе воркера только постфактум, строкой в
`AutonomyRun`. Автопилот работает вслепую, поэтому его страшно включать — и это правильный страх.

**1.3 Стадий в контракте прогресса нет.** `jobs.report(done, total, current)` — счётчик и имя
домена. Воронка скоринга (T0 RD → T1 whois → T2 РКН/блэклист/эхо → T3 Wayback → T3b Ahrefs)
размечена внутри `_funnel()` комментариями, но наружу не выходит. Показать «на каком этапе что
делается» сегодня нечем.

**1.4 Экран M1 — дамп базы, а не рабочее место.** 5827 строк, отсортированных по score, из них
5605 «найден» (сырой фид, делать с ними нечего). 18 доменов, ждущих решения, утонули в этом дампе.

**1.5 Инбокс не отличает «спорный» от «непроверенного» — дыра в гейте курации.**
[scoring.py:38](../../../backend/app/services/scoring.py#L38) не даёт авто-одобрить домен без
успешной проверки истории (гард держит) — домен уходит в `scored`, то есть **в инбокс к человеку**.
Значит: Wayback лежит час → в инбокс сыпется всё подряд вперемешку с реально спорными, и выглядит
там ровно так же — «оценён, score 0.86». Человек одобряет, считая, что машина историю проверила.
Машина её не проверяла. Гейт курации вырождается в штамповку — ровно в том инварианте (домены за
чистую историю), ради которого гейт и существует. Данные есть: `score_breakdown.errors` хранит
`wayback:ConnectError`. Их никто не показывает.

**1.6 Запущенную задачу нечем остановить.** «Проверить весь пул» на 5605 доменов — это часы работы и
сожжённые квоты A-Parser; единственный способ прервать — рестарт контейнера.

## 2. Что делаем

Реестр задач переезжает в PostgreSQL и обрастает стадиями; Пульт становится диспетчерским экраном;
M1 из дампа превращается в очередь работы человека.

**Это волна 1 из трёх.** Макеты на весь фронт панели лежат в [docs/design/](../../design/) (сделаны в
Claude Design, 2026-07-12) — они держат наши токены (холодный синий `#2563c9`, IBM Plex Sans,
сайдбар 200px), поэтому палитру мирить не нужно. Волна 1 берёт из них Пульт, M1 и общий язык
(§6, §8); волны 2–3 — в §10.

**Хард-гейты не трогаются.** Деньги (`confirmed_by_human=true`) и редактура (публикация только из
`edited`) остаются ровно там, где стоят. Пакетное одобрение из §6 — это по-прежнему решение
человека, просто одним кликом вместо восемнадцати; оркестратор к нему не прикасается.

## 3. Реестр задач в PostgreSQL

### 3.1 Модель

Новая таблица `job_run` (миграция `0007_job_run`):

| поле | тип | смысл |
|---|---|---|
| `id` | int PK | |
| `name` | str | `discovery` / `score` / `recheck` / `sweep` |
| `trigger` | str | `manual` (кнопка) / `auto` (воркер) |
| `status` | str | `running` / `done` / `failed` / `cancelled` |
| `stage` | str | ключ текущей стадии (`whois`, `cctld`, `provision`…) |
| `stages` | JSON | снимок степпера: `[{key, label, state}]`, state = `done`/`active`/`pending`/`skip` |
| `done`, `total` | int | счётчик прогресса; `total=0` = неопределённый режим |
| `current` | str | что сейчас в работе (домен / источник) |
| `message` | str | итог прогона («проверено 200: занято 3, свободны 197») |
| `error` | str \| None | текст падения |
| `cancel_requested` | bool | человек нажал «стоп» |
| `started_at`, `updated_at`, `finished_at` | datetime | |

Частичный уникальный индекс `(name) WHERE status='running'` — single-flight **между процессами**:
воркер не сможет запустить второй `score`, пока оператор гоняет свой (сегодня может, и они вдвоём
жгут квоту A-Parser). Индекс поддерживают и PostgreSQL (`postgresql_where`), и SQLite
(`sqlite_where`) — проверено на живом харнессе: DDL выходит частичным, повторный прогон после
`done` замок не держит.

Отдельного параметра `lock=False` у свипа **нет**: индекс срабатывает на любой `running`-строке,
а протухшую (контейнер убили) гасит `_reap()`, поэтому замок не может повиснуть навсегда.
Свой замок свипа (`AutonomyRun._acquire_lock`) остаётся как есть — `AutonomyRun` продолжает быть
историей автопилота, `job_run` — прогрессом.

**Замок, сработавший штатно, — не ошибка.** Если свип дошёл до стадии `score`, а оператор в этот
момент гоняет свой, `track("score")` поднимет `AlreadyRunning` — это ровно то, ради чего замок и
ставили. Такую стадию свип обязан пометить «пропущена, занята ручным прогоном», а не красить весь
прогон в `failed`: иначе наблюдаемость начнёт кричать волком на собственную же защиту.

### 3.2 Контракт `services/jobs.py`

```python
jobs.track(name, *, trigger="manual", stages=None)  # контекст-менеджер: строка в БД,
                                                     # ловит исключение → status=failed
jobs.spawn(name, target) -> bool       # фоновый поток (как сегодня start); False = уже идёт
jobs.report(name, done, total, current="", stage="", message="")
jobs.cancelled(name) -> bool           # сервис спрашивает между итерациями
jobs.request_cancel(name)
jobs.live() -> list[dict]              # все running — для Пульта и шапки
jobs.last(name) -> dict | None         # последний завершённый прогон (итог)
```

**Колбэки `on_progress` из сервисов убираются.** `run_discovery` / `score_pending` /
`recheck_acquirability` / `run_sweep` сами открывают `with jobs.track(...)` и репортят внутрь.
Кто бы их ни позвал — кнопка панели, оркестратор, cron воркера — прогресс появляется бесплатно.
Именно прокидывание колбэка только из панели и делало работу воркера невидимой.

`score_domain` (одиночный домен по кнопке в строке) записи реестра **не** открывает — он
мгновенный.

### 3.3 Терминальный контракт (сохраняется)

Проверять в порядке: `error != None` → упал; `status == 'running'` → идёт; иначе завершён успешно,
**даже если `done == total == 0`** (discovery без кандидатов). `done/total` — отображение, не признак
терминала. Крашнутый контейнер: строка `running` с `updated_at` старше 15 минут — показываем
«задача оборвалась» (переиспользуем порог `STALE_MIN` оркестратора).

**Протухание — это показ, а не мутация.** Чтение (`live` / `progress` / `last`) только считает флаг
`stale` и ничего не пишет. Гасит протухшую строку (`_reap`) **исключительно** попытка занять замок —
`_open()` и `is_running()`. Иначе вышло бы вот что: панель поллит `live()` раз в 1.5 с, а стадия
свипа `generate` (LLM по 5 сайтам) законно молчит дольше 15 минут — и поллер пометил бы **живую**
задачу упавшей, отпустив замок под вторую. Реап на пути чтения — это самострел.

Флаг `stale` считается в Python, поэтому дату надо приводить к tz-aware: SQLite отдаёт `DateTime`
**naive** (проверено), PostgreSQL — aware. Тот же приём уже применён в `acquirability_verdict`.

## 4. Стадии — «на каком этапе что делается»

Форма — **карточка задачи** из макета [02-домены-m1.png](../../design/02-домены-m1.png): синяя
полоса-акцент слева, заголовок «Запуск проверки · ИДЁТ · сейчас: avto-help.ru», справа «✕ Отменить»,
под ним **чипы стадий** (пройденные — `✓` с серой заливкой, активная — залита акцентом, будущие —
контурные), внизу полоса с процентом и остатком.

Чипы, а не стрелочная цепочка: проще верстать и не разваливаются на узком экране.

```
┃ ▶ Запуск проверки  [ИДЁТ]   сейчас: avto-help.ru          [✕ Отменить]
┃ ✓ RD из фида  ✓ whois-возраст  ✓ РКН/блэклист  ✓ эхо в индексе  ● Wayback-история
┃ ▓▓▓▓▓▓▓░░░░░░░░░░░░   34%   34 / 100   осталось 66
```

**Три состояния карточки** ([new-03-карточка-состояния.png](../../design/new-03-карточка-состояния.png)):

1. **Идёт** — активный чип залит акцентом, полоса синяя, справа «✕ Отменить».
2. **Упала** — красная полоса-акцент, бейдж «ОШИБКА», подпись «A-Parser не ответил (timeout) **на
   стадии whois-возраст**», чип этой стадии помечен `✕`, полоса красная и подписана
   **«встала 18 / 100»** (не «ошибка» — видно и *что* сломалось, и *где*), справа «↻ повторить».
3. **Остановлена человеком** — «остановлена на 34 / 100», нейтрально.

У каждой задачи свой набор чипов, отражающий её физику:

- **discovery**: стадии = включённые источники (из `sources_enabled`) + дедуп + запись. `total`
  заранее неизвестен → бар неопределённый, но степпер точный.
- **score**: общий бар = домены N/M; степпер = воронка **текущего** домена. `_funnel()` получает
  отметку стадии (`jobs.report(stage=...)`) в местах, где сегодня стоят комментарии T0–T3b.
- **recheck**: одна стадия (whois), бар N/M.
- **sweep**: стадии = `STAGES` оркестратора (7 ключей → русские лейблы); выключенные тумблером —
  `state=skip`, не исчезают, чтобы было видно, что стадия отключена, а не сломалась.

## 5. Пульт — диспетчерский экран

Сразу под строкой автопилота — блок **«Машина сейчас»** (макеты
[new-01](../../design/new-01-пульт-машина-занята.png) /
[new-02](../../design/new-02-пульт-машина-простой.png)).

**Занята**: карточки задач из §4, **все активные разом**. Свип помечен отдельным чипом
`✈ автопилот` — ручной запуск не должен путаться с cron'овым. Один компонент, один источник данных;
именно здесь впервые видно работу воркера.

**Простаивает**: вместо карточек — сводка последних прогонов с человеческими итогами:

```
Поиск дропов              собрано 4 102 домена                        вчера 14:02
Проверка                  прогнано 100 доменов через воронку               12:40
Перепроверка занятости    занято уже 3 из отобранных → ушли в rejected   3 дня назад
```

Под ней — кнопки быстрого запуска (`↻ Найти дропы`, `▶ Запустить проверку`) прямо с Пульта. Значит
`POST /run/*` должны возвращать оператора туда, откуда он нажал (`?back=` или `Referer`), а не
жёстко на `/domains`.

Тонкая строка того же прогресса — в шапке `base.html`, то есть на **любом** экране: ушёл в
Настройки, пока идёт проверка — всё равно видишь, что машина работает, и можешь остановить.

**Экран Автопилота переходит на тот же компонент.** У `autopilot.html` сегодня свой `#prog` со своим
поллером `/run/sweep/progress` — он умрёт вместе с роутом. Ставим туда `#machine`: свип и есть его
задача, а карточка со стадиями `STAGES` показывает её честнее старой безымянной полосы. После этого
CSS-класс `.progress` не нужен никому и удаляется целиком.

Транспорт — поллинг `GET /api/jobs/live` раз в 1.5 с (как сегодня). SSE не берём: лишняя сложность
ради одного оператора.

**Автоперезагрузка страницы по завершении задачи — только там, где есть `#machine`** (Пульт, M1,
Автопилот): их таблицы и счётчики после прогона протухают. На остальных экранах перезагружать
НЕЛЬЗЯ — свип, дописавшийся в фоне, пока человек вычитывает черновик в `/pages/{id}`, снёс бы
несохранённую редактуру. Гейт редактуры защищает контент от машины; ради наблюдаемости ломать это
нельзя.

## 6. Экран M1 — по макету [02-домены-m1.png](../../design/02-домены-m1.png)

**Шапка.** Заголовок + строка-объяснение под ним («конвейер отбора: поиск дропов → проверка → твоё
решение → покупка руками»), справа статус: «свип 12:40» и чип «ждёт решения 2» (кликабелен — ведёт
в инбокс).

**Воронка плитками** — тот же ряд, что на Пульте (офферы / найдено / на решении / одобрено /
куплено + «подтвердить» из M2). Плитки-гейты помечены точкой акцента. По клику на «отклонено» —
**разбор причин отказа** (блок 4).

**Блок 1 · Действия — одной строкой, а не тремя карточками.**

```
[↻ Найти дропы] [цены]  │  ПРОВЕРИТЬ [5] [▶ Запустить] [весь пул]  │  [⟳ Перепроверить занятость]
● 2 не сверялись 3+ дня                          полный реестр — вкладка «все домены»
▸ что делают эти три действия
```

Три станции схлопываются в одну карточку действий; длинные объяснения (нынешние `<details>` «зачем
это») переезжают в один общий `<details>` внизу карточки. Экран дышит, шильдик сохраняется.

*Расхождение макетов:* [02](../../design/02-домены-m1.png) даёт поле «сколько доменов» + «весь пул»,
[new-04](../../design/new-04-домены-инбокс.png) — только «Проверить пул». Оставляем оба: прогнать
пятёрку на пробу, прежде чем жечь квоту A-Parser на всём пуле, — рабочий сценарий.

Пока задача идёт — под карточкой действий появляется карточка задачи из §4 (чипы стадий + полоса +
«✕ Отменить»).

**Блок 2 · Ждёт твоего решения** (`scored`) — макет
[new-04-домены-инбокс.png](../../design/new-04-домены-инбокс.png).

Строка инбокса: **слева отдельная колонка срочности** («СРОК ДРОПА · ЧЕРЕЗ 2 ДНЯ», янтарная полоса
по левому краю), затем домен с бейджем источника, `score` / `RD` / возраст, вердикт по истории,
справа `[✓ одобрить] [✗ отклонить]`. Срочность стоит первой намеренно: она читается раньше домена и
score.

- **Сортировка по срочности.** Дефолт — `acquire_deadline` по возрастанию (nulls last), не по score:
  домен, дропающийся завтра, стоит выше красивого с дропом через месяц. Над списком — плашка «через
  2 дня дроп: 3 домена» и подпись «сортировка по срочности дропа, а не по score». Переключатель на
  score остаётся.
- **Срочность — это близкий дедлайн, а не наличие дедлайна.** Янтарная полоса ставится только при
  `acquire_deadline <= now + 3 дня`. Если вешать её на любой непустой дедлайн, полосу получат ВСЕ
  backorder-домены (у них дедлайн есть всегда) — и колонка срочности, ради которой всё затевалось,
  перестанет что-либо выделять. Дедлайн приводится к tz-aware перед сравнением (см. §3.3).
- **Пометка «оценён вслепую»** (§1.5). Если `score_breakdown.errors` содержит `wayback:` / `rkn:` /
  `blacklist:` / `searxng:` — вместо «история чистая» строка несёт предупреждение «история НЕ
  проверена: Wayback был недоступен» и кнопку «▶ перепроверить» (повторный `score_domain`). Янтарный
  левый бордер уже занят срочностью, поэтому «вслепую» — плашка **внутри** строки, не бордер.
  Человек больше не штампует непроверенное, думая, что оно проверено.

**Пакетное решение.** Строка над списком: «Одобрить все со score ≥ `[0.80]`» + `[✓ Одобрить пакет]`,
живой счётчик «попадёт **9** доменов» и явная оговорка «⚠ помеченные "вслепую" в пакет не попадают».
Гейт курации остаётся человеческим — это по-прежнему клик человека, просто один вместо восемнадцати.

**Пустой инбокс** ([new-05](../../design/new-05-домены-инбокс-пусто.png)): «Решать нечего — все
оценённые домены разобраны» + кнопки «▶ Запустить проверку» / «↻ Найти дропы». Пустое состояние
объясняет следующий шаг, а не просто молчит.

**Блок 3 · Готовы к выкупу** (`approved`): домен, цена, дедлайн дропа, `[＋ в очередь выкупа]`,
`[🛒 купил руками]`. Мост в M2. Макета нет — верстаем строкой инбокса, меняя правые действия.

**Блок 4 · Разбор причин отказа** — макет
[new-06-причины-отказа.png](../../design/new-06-причины-отказа.png). Открывается **модалкой** по
клику на плитку «отклонено». Причины с полосами и долями (`Мало доноров low_rd — 94 · 50%`,
`Молодой домен too_young — 41 · 22%`, `Грязная история history_dirty — 28 · 15%`, `Занят
not_acquirable — 14`, `Низкий скор low_score — 9`, `РКН — 3`).

Ключевое — **легенда внизу делит причины на две природы**:

> 🟠 режет порог — **144 (76%)**: можно ослабить  🔴 грязь/РКН — не трогать

Это превращает разбор из статистики в решение: видно, сколько отсева — твой выбор порога (`low_rd`,
`too_young`, `low_score`), а сколько — объективная грязь (`history_dirty`, `rkn`, `blacklist`) или
чужая покупка (`not_acquirable`). Рядом — «⚙ настроить пороги →» на `/settings`.

## 8. Баннеры

остаётся.

## 10. Тестирование (офлайн, герметично)

Сейчас на ветке **276 passed**, pyflakes чист — это точка отсчёта.

Новое:

- `job_run`: single-flight — второй `track("score")` при живом первом падает/возвращает False;
  `cancel_requested` → сервис завершается со `status=cancelled` и **частичным** `done` (домены после
  стопа не тронуты).
- Терминальный контракт: `error` проверяется раньше `done/total`; `done==total==0` без ошибки = успех.
- Упавшая задача помнит, **на какой стадии** встала (`stage` переживает `_close(failed)`).
- Стадии: `score_pending` на моках проходит T0→T3b и репортит соответствующие `stage`;
  отсеянный на T1 домен НЕ доходит до Wayback (регрессия воронки сохраняется).
- Прогресс воркера: `run_sweep(trigger="cron")` без всяких колбэков создаёт `job_run` со `stages`
  из `STAGES` — то есть Пульт увидит автопилот.
- Инбокс: домен с `score_breakdown.errors=["wayback:ConnectError"]` помечен «вслепую» и **не
  попадает** в пакетное одобрение.
- Сортировка: домен с ближним `acquire_deadline` и меньшим score стоит выше.
- Разбор причин: счётчики по `reject_reason` сходятся с общим числом `rejected`.
- Хард-гейты: тест, что оркестратор по-прежнему не зовёт `confirm_order` / `mark_edited`.

Переезд (иначе сьют покраснеет — эти тесты сегодня стоят на `/domains?...`):

- `test_web_fixes.py` — кламп лимита, бейдж `reject_reason`, скрытие `not_acquirable`,
  локализованные лейблы → все на `/domains/pool`.
- `test_autopilot_panel.py` — локализованные чипы фильтров → `/domains/pool`.
- `test_pipeline.py` — `?status=purchased` → ссылка на карточку сайта → `/domains/pool`.
- `test_jobs.py`, `test_sources.py`, `test_orchestrator.py`, `test_recheck_acquirability.py` —
  снять `on_progress`.
- **Referer в тестах обязан быть `http://testserver/...`**: CSRF-гард (`main.py:csrf_guard`) отбивает
  403, если хост Referer'а не равен Host. Тест возврата «туда, откуда нажали» с чужим Referer'ом
