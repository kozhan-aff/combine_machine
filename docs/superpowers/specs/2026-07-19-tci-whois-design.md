# Прямой whois TCI для .RU/.РФ/.SU — дизайн

**Запрос оператора:** «все ру домены направим на него» — заменить A-Parser
Net::Whois на прямой whois-сервер координатора зон для российских TLD.

## Проблема

T1 воронки (whois — возраст + занятость) платят ~77% доменов пула: бесплатно
на T0 отсеиваются только `low_rd` + `feed_flag` (23% отказов по живой разбивке
481 отклонённого на боксе 2026-07-19). Whois идёт через A-Parser
(`Net::Whois`, `aparser.py:205`) — секунды-минуты на домен и расход квоты.
Замеренная скорость воронки на живом прогоне: **28 с/домен**, весь пул 5550 —
~1.8 суток.

При этом дата дропа (`free-date`) в whois-ответе ЕСТЬ, но A-Parser её не
отдаёт — отсюда «бездедлайновый пул» (~5.4 тыс. доменов переопрашиваются
вечно и не имеют терминального состояния, см. CLAUDE.md «Что НЕ починено»).

## Живые факты (проба 2026-07-19, `whois.tcinet.ru:43`, сырой TCP)

Формат снят живьём, не по документации — дисциплина проекта после двух
дорогих ошибок (whois-возраст, cctld-дата дропа).

| случай | ответ | вывод |
|---|---|---|
| занятый `usefulscript.ru` | `domain:` / `state:` / `created:` / `paid-till:` / `free-date:` | занят, есть возраст И `free-date` (см. ниже — это НЕ дата дропа) |
| свободный `zzq-nonexistent-9f3k2.ru` | `No entries found for the selected source(s).` | свободен |
| `xn--80aswg.xn--p1ai` (punycode) | полная запись | зона поддержана |
| `сайт.рф` (кириллицей) | **`Invalid request.`** | punycode ОБЯЗАТЕЛЕН |
| `ok.su` | полная запись | зона поддержана |
| **`example.com`** | **`No entries found`** | ⚠️ см. ниже |

**Скорость: 8 запросов за 0.3 с (~37 мс/домен), троттлинга нет.** Без ключа,
без регистрации, stdlib `socket`, никаких зависимостей.

### `free-date` — это ПРОЕКЦИЯ, а не дата дропа (поправка 2026-07-20)

Первая редакция этой спеки назвала `free-date` «датой дропа». **Это неверно**, и
Задача 2 успела унаследовать ошибку — поймано финальным ревью. Живая проба:

| домен | paid-till | free-date |
|---|---|---|
| `yandex.ru` | 2026-09-30 | 2026-11-01 |
| `mail.ru` | 2026-09-30 | 2026-11-01 |
| `usefulscript.ru` (реально истекает) | 2026-06-17 | 2026-07-21 |

`free-date` = `paid-till` + запас реестра, и она есть у **каждого** занятого
домена. У `yandex.ru` она тоже есть — дроп при этом очевидно не случится. Читать
её следует как «освободится, ЕСЛИ владелец не продлит».

Отсюда два правила:

1. **Протухшую проекцию обновляем вперёд.** Домен, дождавшийся своей проекции и
   продлённый владельцем, отвечает whois'ом «занят» + НОВЫЙ `free-date` на год
   вперёд. Выбрасывать этот свежий ответ (правило «заполняем только пустой
   дедлайн») значило судить домен по мёртвой дате и терминально хоронить его в
   `not_acquirable` — на масштабе всего бездедлайнового пула, притом приоритетно
   (просроченный дедлайн стоит в ярусе 1 очереди `score_pending`).
   Условие обновления: `lane != "bid"` И дедлайн просрочен с запасом `DROP_GRACE`
   И новая дата строго дальше старой. Дедлайн из фида, который ЕЩЁ НЕ НАСТУПИЛ,
   не перезаписывается никогда — инвариант «фид авторитетнее» цел. Исключение
   `bid` защищает денежный путь M2: там «дедлайн прошёл, домен занят» означает
   снайпнутый конкурентом домен — законный терминал, а не устаревшая проекция.
2. **Панель не выдаёт проекцию за факт.** `deadline_source` (`"whois_projection"`)
   кладётся в `score_breakdown` — миграция не нужна, тот же приём, что у
   `whois_source`. В инбоксе такая дата подписана «ОСВОБОДИТСЯ*» с пояснением в
   `title`, а не «СРОК ДРОПА». Прецедент — метка «оценён вслепую» и честно
   помеченный мёртвым `authority` для backorder: не притворяемся знающими то,
   чего не знаем.

### Главная ловушка

`example.com` — домен ЗАНЯТ, но TCI отвечает `No entries found`, дословно как
на свободный .ru. Наивная маршрутизация «шлём всё в TCI» пометила бы каждый
не-TCI домен как свободный и утащила бы их в выкуп. Поэтому **зона решает
жёстко, до запроса**: `ru`/`su`/`xn--p1ai` → TCI, всё остальное → A-Parser.
Никакого «попробуем, вдруг ответит».

Третье состояние `Invalid request.` — это НЕ «свободен» и НЕ «занят», а
`available=None` (не определить), тот же контракт, что у A-Parser.

## Решение

Три слоя, по конвенции проекта (`integrations/` = транспорт, логика в
`services/`):

1. **`integrations/whois_tci.py`** — транспорт. `TciWhoisClient.probe(domain)`
   → `{"available": bool|None, "created": datetime|None, "free_date": date|None}`.
   Тот же контракт, что `AParserClient.whois_probe()`, плюс `free_date`.
   Сетевой сбой пробрасывается (как у whois_probe) — судит вызывающий.
   `handles(domain) -> bool` — принадлежность зоне TCI (по punycode-хвосту).
2. **`services/whois.py`** — маршрутизатор. `probe(domain, clients)`:
   зона TCI → `clients["tci"].probe()`; сбой TCI → фолбэк на
   `clients["aparser"].whois_probe()`, факт фолбэка виден в
   `whois_source` возвращаемого словаря (`"tci"` | `"aparser"` |
   `"aparser_fallback"`), не глотается молча.
3. **`services/scoring.py`** — обе точки вызова (`_funnel` T1 строка 436 и
   `recheck_acquirability` строка 889) переходят на маршрутизатор.
   `_clients()` получает ключ `"tci"`.

`sig["whois_source"]` сохраняется в `score_breakdown` — оператор видит, чем
судили домен, и это же попадает в `domain_score_log`.

## Что НЕ делаем

- **Не выбрасываем A-Parser whois.** Не-TCI зоны и фолбэк на нём же. Один
  канал = один отказ роняет T1 целиком.
- **Не трогаем `acquire_deadline` из `free_date` в этой задаче** — это смена
  ПОВЕДЕНИЯ воронки (`acquirability_verdict` судит по дедлайну), а не
  ускорение. Отдельная задача 2 с собственным ревью.
- **Не добавляем кэш/бюджет вызовов.** `max_whois_per_run` остаётся как есть —
  он защищает квоту A-Parser, которая при TCI-маршруте просто не тратится.
  Снимать кап — отдельное решение оператора после живого прогона.
- **Не гоняем TCI для не-TCI зон «на пробу»** — см. ловушку `example.com`.

## Герметичность тестов

Рубильник `_no_live_network` патчит `socket.getaddrinfo` ловушкой-`BaseException`
— сырой TCP TCI из теста упадёт громко, `except Exception` фолбэка его НЕ
поймает (это намеренно: тест обязан подменить клиент явно, а не тихо уехать на
A-Parser). Все существующие фейковые словари клиентов в тестах получают ключ
`"tci"`.

## Тестирование

- Парсер (чистые функции, без сети): занятый / свободный / `Invalid request.` /
  мусор → правильные `available`/`created`/`free_date`.
- Маршрутизатор: `.ru`/`.su`/`.xn--p1ai` идут в TCI; `.com`/`.net` — в A-Parser
  (регресс на ловушку `No entries found`); сбой TCI → фолбэк с
  `whois_source="aparser_fallback"`.
- Кириллический `.рф` нормализуется в punycode до запроса.
- Воронка: домен .ru проходит T1 без единого обращения к A-Parser.
