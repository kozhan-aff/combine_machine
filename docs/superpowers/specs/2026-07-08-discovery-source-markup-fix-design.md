# Discovery-источники cctld/reg_ru/sweb — починка разметки (design)

## Контекст

Аудит M1 (2026-07-07) выключил `cctld`/`reg_ru`/`sweb` в `SOURCES_ENABLED`
(`scoring_config.py`) — живая разметка не была выверена, и живой прогон дал:
- `cctld` — «собирает мусор» (8 «кандидатов» = навигационные ссылки страницы,
  не реестр дропов).
- `reg_ru` — A-Parser-прокси вернул `596 HTTPS proxy error: 502 Bad Gateway` →
  `fetch_html` требует `200` → молча `None` → 0 строк.
- `sweb` — `ReadTimeout` на A-Parser.

Пользователь попросил (1) выверить разметку, чтобы источники реально работали,
и (2) рассмотреть включение `reg_ru` по умолчанию. Ниже — по каждому источнику
живая проверка (прямой `curl`/Python с Mac, без A-Parser и без бокса) и точный
фикс. Все три бага — РАЗНЫЕ, не один общий.

## 1. `cctld.py` — тянули не ту страницу

`https://cctld.ru/service/dellist/` — это лендинг, а не список. Настоящие
данные — в **ежедневных ZIP-файлах**, ссылки на которые лежат на этой же
странице: `RUDelList<YYYYMMDD>.zip` и `RFDelList<YYYYMMDD>.zip`
(напр. `/files/docs/pendingdelete/RUDelList20260708.zip`). Проверено вживую:
внутри — простой текстовый файл, один домен на строку. 9463 `.ru` + 438 `.рф`
строк на день проверки; UTF-8, 0 пустых строк, 0 дублей, 100% строк проходят
форму домена. `.рф`-домены уже в punycode — прогнаны через реальный
`canonical_domain()` из `discovery.py`, идемпотентны (не ломаются повторным
IDNA-кодированием).

**Новая логика `CctldClient.list_dropping()`:**
1. `GET https://cctld.ru/service/dellist/` (как сейчас, прямой httpx —
   реестр без антибота, A-Parser не нужен).
2. Извлечь href обоих zip регэкспом (не строить дату самим — независимость
   от таймзоны/выходных/факта публикации на конкретный день):
   ```python
   _ZIP_HREF = re.compile(r'href="([^"]*(?:RUDelList|RFDelList)\d{8}\.zip)"')
   ```
3. Скачать оба zip (`httpx`, тот же клиент), распаковать в памяти
   (`zipfile.ZipFile(io.BytesIO(content))`, стдлиб), прочитать единственный
   `.txt`-член архива, `splitlines()` — без регэкспа на домен вообще, файл
   уже чист.
4. Оба TLD объединяются в один источник `source="cctld"` (как сейчас).
   `referring_domains` остаётся `None` — источник этой метрики не даёт;
   T0 (`low_rd`) уже спокойно пропускает `NULL` RD.
5. Если один из двух zip не найден/не скачался — не валить весь источник:
   отдать то, что удалось (частичный результат лучше нуля).

`ping()` — считать успехом, если хотя бы один href найден на лендинге
(дешёвая проверка живости, без скачивания zip).

## 2. `regru_drops.py` — точечный якорь вместо общего regex, транспорт → прямой httpx

Живая проверка (2 страницы, включая **голый `httpx.Client()` без кастомных
заголовков** — то есть ровно то, что шлёт `BaseClient`): reg.ru отдаёт `200`
и настоящую HTML-таблицу, никакой анти-бот стены, дефолтный UA
`python-httpx/0.28.1` не блокируется. Прошлая `596 proxy error` — похоже,
разовая проблема ротации прокси на стороне A-Parser, не защита сайта.

**Правило транспорта (уточнено с пользователем):** раз discovery берёт
только страницу 1 (1 запрос на источник за прогон — тот же профиль, что и
`cctld`), тянем **прямым httpx** (`BaseClient`, без A-Parser) — меньше
движущихся частей, ниже риск бана домашнего IP при *редких* прогонах вручную.
A-Parser (ротация прокси) — резерв на случай, если **позже** добавим
пагинацию (десятки/сотни запросов за прогон); тогда транспорт нужно будет
пересмотреть обратно на A-Parser. Пока пагинации нет — A-Parser не нужен.

Текущий парсер (`_parse_domains` из `cctld.py`, общий regex «любой
домен-подобный токен в тексте страницы») на HTML ловит мусор: `reg.ru` (54
раза, свой домен в разметке), `yandex.ru`, `rbc.ru`, `ok.ru`, `gov.ru`,
`dzen.ru`, `cctld.ru` — из навигации/футера, вперемешку со штучными реальными
строками таблицы.

**Точный маркер (проверено на 2 страницах, 38/38 и 38/38 совпадений, 0
мусора):** ячейка с доменом — единственная с классом `b-table__cell_node_first`
(ячейки с датами — тот же базовый класс, но без `node_first`; заголовок
таблицы «Домен» — это `<th>`, не `<td>`, регэксп на `</td>` его не подхватывает):

```python
_DOMAIN_CELL = re.compile(r'b-table__cell_node_first">\s*([^<]*?)\s*</td>')
```

Транспорт — прямой httpx (`BaseClient`), как cctld — см. правило транспорта
выше: 1 страница = 1 запрос за прогон, A-Parser не нужен для такого объёма.

**Только страница 1** (`_URL` без `?page=`) — витрина постраничная (~15664
доменов всего), пагинацию за пределы первой страницы сознательно не делаем
(см. «Не делаем» ниже).

## 3. `sweb_drops.py` — якорь на пару label+value, НЕ на общий класс

**Важно (найдено при повторной проверке, до кода):** в отличие от reg.ru, на
sweb.ru класс `domains-deleted__text` **общий** для домена И обеих дат
(регистрация + освобождение) — нет отдельного «первая колонка» маркера,
как у reg.ru. Наивный якорь на этот класс дал бы 30 совпадений на 10 реальных
доменов на странице, включая строки вида `07.07.2026` — а такая строка
**прошла бы** обычную проверку «похоже на домен» (три группы через точки) и
попала бы в базу как кандидат-домен.

**Исправленный якорь** — берём `domains-deleted__text` только там, где
непосредственно предшествующий `domains-deleted__label` — это буквально
«Домен» (проверено: 10/10 чистых доменов на странице, `total item blocks == 10`):

```python
_DOMAIN_FIELD = re.compile(
    r'domains-deleted__label">Домен</span>\s*'
    r'<span class="domains-deleted__text">\s*([^<]*?)\s*</span>'
)
```

Транспорт — прямой httpx (как reg_ru, то же обоснование: 1 запрос за прогон).
Только страница 1 (`?page=1`, витрина глубоко постраничная — до `?page=1567`
на день проверки).

## Дефолты источников — без изменений

`SOURCES_ENABLED` остаётся `{backorder: True, cctld: False, reg_ru: False,
sweb: False}`. Пользователь включит источники руками через `/settings` после
живого прогона на боксе (офлайн-тесты покрывают код мок-данными, но
реальность A-Parser-прокси/живой разметки может отличаться — только живой
прогон это подтвердит).

## Тестирование

Офлайн, без сети — фикстуры из реально скачанных сэмплов (урезанные до
необходимого минимума: сама таблица/список + немного окружающего мусора,
чтобы тест доказывал фильтрацию, а не только happy path):
- `cctld`: фрагмент лендинга с zip-ссылками + маленький синтетический
  zip/txt (обе TLD).
- `reg_ru`: фрагмент HTML с 2-3 реальными строками таблицы + шумовые ссылки
  (`reg.ru`, `yandex.ru` в другом месте разметки) — тест должен доказывать,
  что шум НЕ попадает в результат.
- `sweb`: фрагмент с 2-3 `domains-deleted__item` блоками (домен + обе даты)
  — тест должен доказывать, что даты (`DD.MM.YYYY`) НЕ попадают в результат,
  это ровно тот баг, что нашли при ревью.

Мокать транспорт (`httpx`) через monkeypatch, как уже принято в проекте
(`backend/tests/conftest.py` режет живую сеть по умолчанию) — сами тесты
валидируют только парсинг. `AParserClient` для этих трёх источников больше
не нужен вообще — `regru_drops.py`/`sweb_drops.py` перестают импортировать
`app.integrations.aparser`.

## Не делаем (осознанно, non-goals)

- Пагинацию за пределы страницы 1 для reg_ru/sweb — **если добавим позже,
  тогда же вернуть транспорт на A-Parser** (правило из секции 2: много
  запросов за прогон → нужна ротация прокси).
- Починку транзиентных A-Parser-проблем (502/timeout) — они больше не
  влияют на эти три источника после перехода на прямой httpx.
- Включение источников по умолчанию в этом же изменении.

## Риски / что не проверено с Mac

- Разметка сайтов может измениться позже — регэкспы завязаны на конкретные
  CSS-классы (`b-table__cell_node_first`, `domains-deleted__text`/`__label`),
  не на семантику. Если сайт обновит вёрстку, парсер снова начнёт давать 0 или
  мусор — тот же класс отказа, что уже был; следующий живой прогон на боксе
  (или `/diag`-подобный контроль по аналогии с blacklist control-probe) это
  покажет.
- Живая проверка делалась с Mac (прямой `curl` и голый `httpx.Client()`), не
  с бокса — домашний IP бокса теоретически может получить другой ответ
  (гео/анти-бот эвристики бывают IP-зависимы), хоть это и маловероятно для
  голого GET без JS-рендера. Живой прогон на боксе подтвердит окончательно.
