# optimizator.ru — референс API (регистрация свободных доменов)

> Реселлерская прослойка над **Руцентр (nic.ru / RU-CENTER)** и **Рег.Ру (reg.ru)**.
> Назначение в проекте: **M2 Acquisition** — регистрация УЖЕ СВОБОДНЫХ доменов оптом
> (гарантия, в отличие от backorder-ставки на дроп). См. [BUILD_SPEC §6.5](../../BUILD_SPEC.md),
> клиент — `backend/app/integrations/optimizator.py`.
>
> Вся дока ниже снята дословно со страниц `optimizator.ru/texts/*.html` (кодировка windows-1251),
> проверено HTTP-запросами 2026-07-05. Что не подтверждено доками — помечено **UNVERIFIED**.

---

## Назначение
API рассчитан на «владельцев хостингов», которые регистрируют/продлевают домены клиентов через
Руцентр по оптовым ценам. Для нашего проекта это способ **гарантированно зарегистрировать
уже освободившийся (свободный) домен** на подконтрольную анкету RU-CENTER.

**Ключевое ограничение модели:** регистрация идёт на **анкету клиента RU-CENTER (`XXX/NIC-D`),
переданную под управление optimizator** («смена партнёра»). Нельзя зарегистрировать домен на
произвольный `nicd` — анкета сначала должна быть переведена под их управление (см. Готчи).

---

## Base URL
```
http://optimizator.ru/?a=api&sa=<METHOD>&<params>
```
- В доках везде **http**. HTTPS отвечает тем же контентом (проверено), но канонично документирован http.
  Для нашего клиента можно использовать `https://optimizator.ru` (безопаснее — api_key в URL). **UNVERIFIED**, что https officially supported.
- Общие параметры для всех методов: `a=api` (константа), `sa=<имя метода>`.
- Запросы принимаются методами **GET или POST** (для всех методов).
- Формат ответа — **JSON-массив** (даже для одного объекта), напр. `[{...}]`.

---

## Auth
- Единственный секрет — **`api_key`** (передаётся параметром в каждом запросе).
- Ключ выдаётся после регистрации аккаунта; находится в ЛК → «Настройки профиля».
- **`nicd` — это НЕ аутентификация**, а данные (номер анкеты RU-CENTER `XXX/NIC-D`), нужен только
  методам `reg_domains` и `check_nicd`.
- IP-whitelist / подпись запроса в доках **не упоминаются** (не подтверждено, что их нет — **UNVERIFIED**).
- Маппинг на `.env` проекта: `OPTIMIZATOR_API_KEY` → `api_key`, `OPTIMIZATOR_NICD` → `nicd`.

### Кодировка (`enc`)
- По умолчанию значения `domain`/`domains` считаются в **CP1251**.
- Если передаёте кириллический IDN (напр. зона `.рф`) в **UTF-8** — добавьте `enc=utf8`.
- Для ASCII-доменов (латиница) параметр `enc` не влияет.

## Методы

Всего 7 методов. Сводка:

| `sa=` | Назначение | Ключевые параметры | Ответ |
|---|---|---|---|
| `balance` | баланс счёта | `api_key` | `[{"balance":1234.56}]` |
| `prices` | цены рег./продления | `api_key`, `domain` (опц.) | `[{"domain","price_registration","price_renewal"}]` |
| `check_domain` | можно ли **продлить** домен | `api_key`, `domain`, `enc?` | `[{"data_end","domain"}]` |
| `check_nicd` | анкета под нашим управлением? | `api_key`, `nicd` | `[{"nicd":"11/NIC-D"}]` |
| `reg_domains` | **регистрация** доменов | `api_key`, `nicd`, `domains`, `enc?` | `[{"order_id":N}]` |
| `renew_domains` | продление доменов | `api_key`, `domains`, `enc?` | `[{"order_id":N}]` |
| `check_order` | **статус заказа** | `api_key`, `order_id` | `[{"order_id":N,"state":"completed"}]` |

Для нашего пайплайна значимы: **`reg_domains`** (регистрация) + **`check_order`** (статус),
вспомогательно — `balance`, `prices`, `check_nicd`.

### 1. `balance` — баланс счёта
Источник: [texts/8.html](https://optimizator.ru/texts/8.html)

```
http://optimizator.ru/?a=api&sa=balance&api_key=abcdef12345678901234567890
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |

**Ответ:**
```json
[{"balance":1234.56}]
```
- `balance` — дробное (до 2 знаков), остаток средств на счёте. Проверять перед `reg_domains`.

---

### 2. `prices` — цены на регистрацию и продление
Источник: [texts/9.html](https://optimizator.ru/texts/9.html)

```
http://optimizator.ru/?a=api&sa=prices&api_key=abcdef12345678901234567890&domain=ru
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `domain` | Нет | домен ПЕРВОГО уровня (зона), только для которого показать цены. Без него — все зоны. |

**Ответ:**
```json
[{"domain":"RU","price_registration":85,"price_renewal":89},{"domain":"xn--p1ai","price_registration":88,"price_renewal":89}]
```
- `domain` — зона (RU, COM, `XN--P1AI` для .рф).
- `price_registration` — цена регистрации; **`0` или отсутствует → зона не регистрируется**.
- `price_renewal` — цена продления; **`0` или отсутствует → не продлевается**.
- (Числа в примере — иллюстративные/старые. Актуальные тарифы с главной 2026-07: Руцентр RU 179/199 руб.,
  РФ 179/199, SU 329/375, COM 1899/2099, NET 1999/2399, ORG 1770/1999; Рег.Ру RU/РФ 179/195.
  Первое число — регистрация, второе — продление.)

### 3. `check_domain` — можно ли ПРОДЛИТЬ конкретный домен
Источник: [texts/10.html](https://optimizator.ru/texts/10.html)

> ВНИМАНИЕ: несмотря на имя, это **НЕ проверка «свободен ли домен для регистрации»**.
> Метод проверяет, можем ли мы **продлить** уже зарегистрированный домен (т.е. под нашим ли
> управлением его анкета). Для проверки свободен ли домен под регистрацию — API метода НЕТ,
> проверяйте сами через whois (см. Готчи и flow регистрации).

```
http://optimizator.ru/?a=api&sa=check_domain&api_key=01234567890abcdefg01234567890&domain=optimizator.ru
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `domain` | Да | домен, который нужно продлить |
| `enc` | Нет | `utf8`, если `domain` — кириллица в UTF-8 (иначе CP1251) |

**Ответ (если продление возможно):**
```json
[{"data_end":"02.12.2016","domain":"OPTIMIZATOR.RU"}]
```
- `domain` — имя домена; `data_end` — дата окончания регистрации (ДД.ММ.ГГГГ).
- Если продлить нельзя — **«будет выведено сообщение об ошибке»** (формат ошибки не документирован, **UNVERIFIED**).

### 4. `check_nicd` — анкета клиента под нашим управлением? (проверка смены партнёра)
Источник: [texts/11.html](https://optimizator.ru/texts/11.html)

```
http://optimizator.ru/?a=api&sa=check_nicd&api_key=abcdef12345678901234567890&nicd=11
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `nicd` | Да | номер анкеты клиента у RU-CENTER (`XXX/NIC-D`) |

**Ответ:**
```json
[{"nicd":"11/NIC-D"}]
```
- Возврат `nicd` → анкета **под управлением optimizator**, на неё можно регистрировать.
- Если анкета не под управлением — **сообщение об ошибке** (формат **UNVERIFIED**).
- **Обязательный предварительный шаг перед `reg_domains`.**

---

### 5. `reg_domains` — РЕГИСТРАЦИЯ доменов  ★ основной метод
Источник: [texts/12.html](https://optimizator.ru/texts/12.html)

```
http://optimizator.ru/?a=api&sa=reg_domains&api_key=abcdef12345678901234567890&nicd=11&domains=optimizator.ru%20segin.ru&enc=utf8
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `nicd` | Да | анкета RU-CENTER (`XXX/NIC-D`), НА КОТОРУЮ регистрируются домены (должна быть под управлением — см. `check_nicd`) |
| `domains` | Да | домены через **запятую, точку с запятой или пробел**, **максимум 30 за раз** |
| `enc` | Нет | `utf8`, если в `domains` кириллица в UTF-8 (по умолчанию CP1251) |

**Ответ (успешная постановка заказа):**
```json
[{"order_id":1}]
```
- `order_id` — целое, номер заказа; статус узнавать через несколько минут методом `check_order`.
- **Один `order_id` на весь вызов** (на всю пачку до 30 доменов), не по одному на домен.
  Как отражается частичный успех пачки в `check_order` — **UNVERIFIED** (см. Открытые вопросы).
- Если домены зарегистрировать нельзя — **«сообщение об ошибке»** (формат **UNVERIFIED**).
- Примечание: на странице метода в описании ответа опечатка — написано «заказ на **продление**
  доменов», хотя это регистрация. Игнорировать.

### 6. `renew_domains` — продление доменов
Источник: [texts/13.html](https://optimizator.ru/texts/13.html) (в нашем проекте пока не используется)

```
http://optimizator.ru/?a=api&sa=renew_domains&api_key=01234567890abcdefg01234567890&domains=optimizator.ru%20segin.ru
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `domains` | Да | домены через запятую/точку с запятой/пробел, **максимум 100 за раз**; все домены — одной анкеты `XXX/NIC-D` |
| `enc` | Нет | `utf8` для кириллицы в UTF-8 |

**Ответ:** `[{"order_id":1}]` (аналогично `reg_domains`, статус через `check_order`).
Примечание: у `renew_domains` `nicd` в параметрах НЕ указан (в отличие от `reg_domains`) —
анкета определяется по самим доменам.

### 7. `check_order` — ПРОВЕРКА СТАТУСА ЗАКАЗА  ★
Источник: [texts/16.html](https://optimizator.ru/texts/16.html) (это и есть метод, на который ссылается BUILD_SPEC)

```
http://optimizator.ru/?a=api&sa=check_order&api_key=abcdef12345678901234567890&order_id=1
```
| Параметр | Обяз. | Значение |
|---|---|---|
| `api_key` | Да | ключ API |
| `order_id` | Да | номер заказа из ответа `reg_domains`/`renew_domains` |

**Ответ:**
```json
[{"order_id":1,"state":"completed"}]
```
- `order_id` — целое, номер заказа.
- `state` — строка. Документировано **единственное** значение: **`completed`** — «заказ выполнен».
- **Значения для НЕЗАВЕРШЁННОГО/ПРОВАЛЬНОГО заказа в доках НЕ приведены** (нет `pending`/`failed`/`error`
  и т.п.). Это главный пробел — см. Готчи и Открытые вопросы. **UNVERIFIED**, как API отдаёт неуспех.

## Поток регистрации (M2)
По методичке optimizator ([texts/15.html](https://optimizator.ru/texts/15.html)) + наш гейт выкупа:

1. **(наш гейт)** заказ отправляется провайдеру ТОЛЬКО при `confirmed_by_human=true` (CLAUDE.md §правило 2).
2. `prices` — убедиться, что цена регистрации нужной зоны задана (`price_registration>0`).
3. `balance` — хватает ли средств на счёте.
4. **Проверить, свободен ли домен, СВОИМИ силами (whois)** — у API нет метода проверки свободности
   под регистрацию (`check_domain` — про продление, а не про свободность).
5. `check_nicd` — анкета `nicd` под управлением optimizator (иначе домен зарегистрировать не на что).
6. `reg_domains` (≤30 доменов) → получить `order_id`, сохранить в `acquisition_orders.provider_order_id`.
7. Подождать **~15 минут** («через несколько минут» на странице метода; «минут через 15» в методичке).
8. `check_order(order_id)` (поллинг) → `state=="completed"` ⇒ успех ⇒ `domains.status=purchased`.
9. **Верификация неуспеха** (т.к. failure-state не документирован): если за таймаут `state != completed` —
   подтверждать whois-ом (домен реально зарегистрирован на нас?) и/или из ЛК; помечать заказ `failed`.

## Лимиты и деньги
- `reg_domains`: **≤ 30 доменов за вызов**. `renew_domains`: ≤ 100 (все одной анкеты).
- Статус заказа готов **не сразу** — опрашивать `check_order` спустя ~15 мин (не долбить сразу).
- Регистрационный сбор списывается с баланса (см. `prices`/`balance`). Актуальные тарифы RU/РФ ≈ 179 руб.
  регистрация. Беречь баланс/квоты (конвенция проекта). Один `order_id` может покрывать до 30 доменов.
- Rate limit по частоте запросов в доках не указан (**UNVERIFIED**).

## Готчи
- **Анкета под управлением — обязательна.** `reg_domains` регистрирует на `XXX/NIC-D`, ПЕРЕДАННУЮ под
  управление optimizator («смена партнёра» в ЛК Nic.Ru). Без этого — ошибка. Наш `OPTIMIZATOR_NICD`
  ДОЛЖЕН быть такой подконтрольной анкетой. Проверять `check_nicd` перед регистрацией.
- **Нет API проверки «свободен ли домен».** `check_domain` — про возможность ПРОДЛЕНИЯ, а не свободность.
  Свободность под регистрацию проверяем сами (whois). Название метода вводит в заблуждение.
- **Не документирован признак ПРОВАЛА заказа.** `check_order.state` документирован только как `completed`.
  Как выглядит зависший/ошибочный заказ — неизвестно. Обязательно закладывать таймаут поллинга +
  fallback-проверку whois/ЛК, чтобы не считать «не completed» вечно «в процессе».
- **Формат ошибок не специфицирован.** При невозможности операции «выводится сообщение об ошибке» —
  не факт что JSON-массив `[{...}]`. Клиент должен: если тело не парсится как ожидаемый JSON-массив
  с нужными полями — трактовать как ошибку и логировать сырой ответ.
- **Один `order_id` на пачку до 30 доменов.** Частичный успех (часть доменов зарегана, часть нет)
  в рамках одного заказа никак явно не отражён в доке — верифицировать по факту (whois по каждому домену).
- **Кодировка по умолчанию CP1251.** Для `.рф`/кириллицы в UTF-8 — обязательно `enc=utf8`.
- **api_key в URL (query).** Использовать https и не логировать полный URL с ключом.
- **`renew_domains` без `nicd`, `reg_domains` с `nicd`** — не перепутать сигнатуры.

## Открытые вопросы / низкая уверенность
1. **Как API отдаёт НЕуспешный/незавершённый заказ** в `check_order`? (`state` = `pending`/`failed`/
   `error`/что-то ещё, или просто отсутствует запись?) — **главный пробел**, влияет на детект провала.
2. **Формат сообщения об ошибке** для `reg_domains`/`check_nicd`/`check_domain`/`balance` — JSON с полем
   `error`? plain text? HTTP-код? — не документировано.
3. **Частичный успех пачки** `reg_domains`: как узнать, какие из ≤30 доменов зарегистрированы, если
   часть провалилась? `check_order` возвращает один `state` на заказ.
4. **Reg.Ру vs RU-CENTER через API.** `nicd` — формат RU-CENTER (`XXX/NIC-D`). Можно ли регистрировать
   через reg.ru-плечо по API и как выбрать регистратора — из доки неясно.
5. **HTTPS официально?** Отвечает, но доки только про http. Rate limits / IP-whitelist — не указаны.
6. Все примеры (`order_id:1`, ключи `abcdef...`) — иллюстративные; реальные значения из ЛК.
